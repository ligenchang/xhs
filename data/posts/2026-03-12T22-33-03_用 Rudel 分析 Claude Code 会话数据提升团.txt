【标题】
用 Rudel 分析 Claude Code 会话数据提升团队效率和质量

======================================================================
【正文】
大家好，我是 Lucy，来自小红书的大模型观测站。最近我在和团队一起用 Claude Code 写项目的时候，发现自己一直在重复同样的操作，却很难判断哪次会话真的有效果，哪次只是浪费时间。于是我们决定自己动手做一个小工具，来把这些零散的会话记录变成可看的数据，这就是 Rudel 的由来。下面我就把我们在使用和开发过程中遇到的一些情况，以及怎么把它跑起来，简单说说，希望对正在用 Claude Code 的同学有点帮助。

起初我们只是想看看自己每天到底花了多少精力在和 Claude Code 打交道。于是我们在本地跑了一个简单的脚本，把每次会话的开始时间、结束时间、还有中间的交互次数都记下来。没想到数据一出来就挺惊讶的：我们总共收集了 1573 个真实的 Claude Code 会话，累计使用了超过 1500 万个 token，产生了 27 万多次交互。这个量虽然不算 huge，但已经足够看出一些趋势了。比如我们发现，其实只有大约 4% 的会话里会真的用到一些高级的技巧或者特定的命令，剩下的大部分时间都是在做些比较基础的事情，比如改改注释或者跑跑测试。还有相当一部分会话，大约有 26%，是在开始不到一分钟就被放弃了，看来很多时候我们可能只是随手点了个指令，然后发现不太对就直接关掉了。

除了这些基本的使用情况，我们还观察到会话的成功率和任务类型关系很大。比如写文档、生成注释这类需求，完成度往往最高；而做重构、或者需要跨文件修改的任务，成功率反而会低一些。有趣的是，我们在会话的前两分钟里经常能看到某种错误的连锁反应——一开始如果出现了一个小误会，后续的几轮交互往往会把这个误会放大，最后导致整个会话被放弃。这个模式虽然不是百分百准确，但只要在前两分钟看到错误开始累积，就能有相当的把握猜测这个会话可能会夭折。至于文章里提到的“没有 meaningful bench”其实是说我们还没有找到一个普遍适用的基准来衡量什么是“好”的会话，不同团队、不同项目可能还有自己的标准。

了解完这些发现以后，我们就开始思考怎么把这些洞察变成实际能用的功能。于是 Rudel 诞生了，它本质上是一个围绕 Claude Code 会话的 analytics 层。你只要装好对应的 CLI，然后在会话结束的时候自动把完整的 transcript 发送到我们的后台，后台会把这些原始文本存进 ClickHouse，再做一些聚合和可视化。这样你就能在仪表盘里看到 token 的使用趋势、会话时长的分布、还有不同模式下的活跃度等等。这些指标对我们来说很有用，比如可以一眼看出哪几天我们在写测试的时候花了最多的时间，或者哪种类型的任务在我们这里通常会花更久。

接下来聊聊怎么把 Rudel 装上去。首先你需要装好 Bun 运行时，这是因为我们的 CLI 是基于 Bun 构建的，装好之后就可以直接用 npm 安装全局的 rudel 包。命令行里执行 `npm install -g rudel` 就行，安装完成后记得先运行 `rudel login`，这一步会打开浏览器让你去授权你的 rudel.ai 账号，授权成功之后就会在本地保存一个 token，后续的操作都会自动带上这个 token。登录之后，你只需要执行一次 `rudel enable`，CLI 会在你的系统里注册一个钩子，这个钩子在 Claude Code 退出的时候会被触发，负责把本次会话的所有数据打包上传到我们的服务器。整个过程是完全自动的，你不需要在每次结束的时候手动去触发什么东西。

如果你之前已经积累了一些会话的文件，也不用担心。rudel 提供了一个交互式的上传命令 `rudel upload`，运行它以后会弹出一个文件选择器，你可以一次性选中多个会话的 transcript 文件，然后它会依次把这些文件发送到后台。这样即使你之前没有开启自动上传，也能把历史数据补进去，后续的分析就能包括这些旧的会话了。上传的过程中你会看到进度条，而且如果有任何文件格式不符合预期的情况，CLI 也会给出相应的提示，帮助你快速定位问题。

有时候你可能不是一个人在使用 Claude Code，而是和几个同学一起协作。这时候 rudel 也支持组织邀请的功能。你登录到 rudel.ai 的网页版，进入 Settings → Organization 页面，填入同事的邮箱，系统会生成一个邀请链接，把这个链接发过去，对方点进去完成注册以后就自动加入到你的组织里了。加入后组织成员的会话数据会在同一个仪表盘里汇总显示，这样你就能很方便地比较不同成员在不同任务类型上的表现，或者找出团队整体上的瓶颈。

关于 rudel 背后是怎么工作的，其实也不算太复杂。只要你开启了 `rudel enable`，CLI 会在你的机器上监听 Claude Code 的退出事件。一旦检测到会话结束，它就会读取当时保存的完整 transcript——这包括了你发给模型的每一个 prompt、模型返回的每一段响应，以及中间工具调用的输出、文件内容、还有可能出现的命令行日志。这些原始数据会被打包成一个 JSON 对象，然后通过 HTTPS 发送到我们的后端服务器。后端其实就是一个 ClickHouse 集群，专门用来存这类高吞吐的事件数据。数据落库之后，我们会跑一些预处理的 SQL 作业，把原始文本切分成 token、统计会话时长、计算交互频率、识别子代理的使用情况等等，最后把这些聚合结果写进另一套表里，供前端的仪表盘读取。整个链路设计上尽量保持低延迟，一般情况下从会话结束到你看到更新的数据也就是几秒的事情。

至于会话里到底会收集哪些信息，官方文档里已经列得很清楚了。每一次上传的会话都包含：会话的唯一 ID、开始和结束的时间戳、你的用户 ID（如果你登录了的话）以及所属的组织 ID、项目所在的路径和包名称、Git 上下文（比如所在的仓库、当前的分支、最近的提交 SHA 和远程地址）、完整的会话 transcript（即所有 prompt 和 response 的原文），还有你在会话中可能调用的任何 sub-agent 的使用记录。这些字段基本上覆盖了你和 Claude Code 交互过程中可能产生的所有痕迹。换句话说，如果你的代码里有敏感信息、或者你在终端里不小心泄露了某种密钥、或者你复制了某个内部 URL，这些都有可能被原样记录进 transcript 里。因此在决定是否开启 rudel 的时候，你需要确认自己所在的项目和环境是否允许把这些数据上传到外部服务。如果你只是在自己的个人玩具项目上随便玩玩，那大概率没有问题；但是如果你在处理公司内部的专有算法或者包含客户数据的代码库，那就得三思而后行了。

即便我们提供的是托管服务（也就是你可以直接去 rudel.ai 注册使用），我们也做了一些隐私上的隔离。后端服务器本身只负责接收和存储这些原始数据，我们的工程团队在正常情况下是无法读取这些 transcript 内容的——也就是说，即便数据躺在我们的硬盘里，也没有人会去把它们当成普通的文件来查看。只有当你自己在仪表盘里查看自己的会话时，系统才会解密并展示对应的数据。这一点在我们的隐私政策里有说明，建议大家在正式使用之前先去看看那份文件，确认自己对数据处理方式完全清楚之后再决定是否开启上传。

对于想要自己搭建私有实例的团队来说，rudel 也提供了自行部署的方案。你可以参考仓库里的 docs/self-hosting.md 文件，里面有详细的步骤说明，从准备数据库、配置对象存储（如果你想把大文件分离出来）到启动后端服务、设置反向代理，都有示例命令和注意事项。基本上只要你有一台能够运行 ClickHouse 和 Node.js 环境的服务器，就可以把后端跑起来。另外还有容器化的方案，如果你习惯用 docker-compose，也可以直接拉我们提供的 compose 文件，一键启动所有必需的服务。自建之后你就完全掌握数据的存放位置和访问权限，适合对数据安全有特别要求的场景。

在开发方面，如果你想为 rudel 贡献代码或者修改某些功能，仓库里的 CONTRIBUTING.md 有完整的指南。它说明了如何在本地编译 CLI、怎么运行测试套件、以及提交 PR 时需要遵循的代码风格检查。我们也有一个专门的 SECURITY.md 文件，里面列出了报告安全漏洞的渠道，强烈建议不要在公开的 issue 里直接贴出可能存在的安全细节，而是按照那里说的方式私下联系我们，这样我们才能及时处理并避免风险扩散。最后，rudel 的代码采用的是 MIT 许可证，这意味着你基本上可以自由地使用、修改和再分发，只要保留原始的版权声明即可。

说完了这么多，我想说其实工具只是一个辅助，真正能让我们在使用 Claude Code 时变得更高效的，还是我们自己对工作流的反思和改进。通过 rudel 看到的那些数字，只不过是一面镜子，帮助我们注意到哪些地方在浪费时间，哪些地方其实还有提升的空间。如果你也在日常里频繁和 Claude Code 打交道，不妨试着装上 rudel，跑一两周看看自己的数据到底长什么样子，也许会有一些意外的发现。好了，今天的分享就到这里，如果你在使用过程中遇到任何问题或者有别的想法，欢迎在评论区留言，我会尽量回复。祝大家编码愉快，下次见！

大家好，我是 Lucy，来自小红书的大模型观测站。最近在团队里使用 Claude Code 时，发现自己总在重复同样的操作，却很难判断哪次会话真的有效果。于是我们自己动手做了 Rudel，把零散的会话记录变成可看的数据，帮助团队看到真实的使用情况。

我们收集了 1573 个真实的 Claude Code 会话，累计超过 1500 万 token，产生 27 万次交互。数据显示只有约 4% 的会话使用了高级技巧，而 26% 的会话在开始不到一分钟就被放弃。成功率与任务类型强相关——写文档、生成注释完成度高，重构跨文件修改则较低。

Rudel 作为 Claude Code 的 analytics 层，只需安装 Bun 并运行 `npm install -g rudel`，接着 `rudel login` 授权后执行 `rudel enable` 即可自动捕获会话 transcript 上传到后台。历史文件也能通过 `rudel upload` 补齐，组织邀请功能让团队数据在同一仪表盘汇总，便于比较成员表现和发现瓶颈。

亮点在于全链路低延迟：会话结束后几秒即可在仪表盘看到 token 趋势、时长分布等指标；数据存放在 ClickHouse，前端只在用户自行查询时解密展示，隐私得到隔离。局限是需要上传完整 transcript，若代码中包含敏感信息需谨慎；此外，目前还没有普遍适用的“好”会话基准，不同团队仍需自行解读。整体趋势是观测层成为大模型开发的标配，帮助团队从经验决策转向数据驱动。

如果你也在频繁使用 Claude Code，不妨试装 Rudel 跑一两周，看看自己的会话数据长什么样子。欢迎在评论区分享你发现的意外模式或使用中遇到的问题，我会尽量回复。
