大家好，今天跟大家聊聊最近在 AI 社区里挺火的一个话题，那就是 Gemini API 的免费额度到底有多少，还有 OpenRouter 那边的免费 tier 能不能当作测试平台使用。最近我在 Twitter 上看到一条比较有意思的消息，里面提到 Google Gemini API 还提供了相当可观的免费额度，而且 OpenRouter 也提供了 free tier 供大家试用。下面我把自己的一些实测感受和一些数字整理出来，供大家参考。

根据那条 tweet，日期是 2026 年 3 月 10 日 06:40 UTC，作者提到 Gemini API 的免费配额是每月 5 万次请求，另外 OpenRouter 的 free tier 可以支撑大约 1 万次调用。这个数字看起来在开发者社区里挺受关注的，尤其是对于那些想要在项目里尝试多模态模型但又不想花钱的同学来说，免费额度的多少直接决定了能不能先行试验。我自己也基于这条信息，搭了一个小实验环境，用 Gemini API 来做文字生成和图像识别的混合演示，结果还算顺利，只是在并发请求数上碰到了一些限制。

 Gemini API 的免费 tier 实际上分为几个层级，最基础的那层叫 “tier 1”，它每月只给 5 万次请求，每次请求的费用是 0，但有一个限制是每秒并发请求数不能超过 10 次。这个限制在实际使用中会导致如果你想一次性跑大批量数据，需要排队或者分批请求。除此之外，tier 1 还提供了 1000 小时的计算资源，用于运行模型推理，这对于一些轻量级的实验来说已经足够。相对的，tier 2 则把免费请求次数提升到 20 万次，并发上限提升到 30 次，同时额外送出 2000 小时的计算资源。不过 tier 2 的可用地区比较有限，只有北美、欧洲和亚太地区的部分节点可以使用，这意味着如果你在中国大陆直接访问，可能会遇到 latency 较高的情况。

 OpenRouter 的 free tier 则相对更开放，它不限制请求次数，而是采用 “按量计费但前 1 万次免费” 的模式。也就是说，你可以免费调用最多 1 万次，超出后才需要付费。这个设计对开发者来说挺友好的，因为你可以在不担心配额耗尽的情况下，随意尝试不同的模型。OpenRouter 还提供了统一的 API 接口，让你可以在不同模型之间切换而不需要改代码，这点在多模型实验中非常有用。不过，OpenRouter 的免费 tier 也有它的小缺点，那就是在高峰期可能会出现排队等待，尤其是当你同时有多个请求在抢占同一资源池时，可能需要等待几秒甚至更长时间才能得到响应。除此之外，OpenRouter 对于某些模型的访问权限是受限的，需要你先申请才能使用。

在实际使用过程中，我发现 Gemini API 的响应速度整体还是比较快的，尤其是在文字生成方面，它的输出质量比很多开源模型要好，语义连贯性比较高。但图像相关的功能因为免费 tier 的限制，只能在每月 5 万次请求的范围内使用，如果你想做大规模的图像标注实验，可能需要先做好请求配额的规划。我自己在实验中采用了批量请求的策略，把需要处理的图像分成几批，每批不超过 5000 次请求，这样就能在不超出配额的情况下完成全部处理。为了避免触发并发限制，我把请求间隔控制在 0.2 秒左右，这样既能保持较好的吞吐量，又不会触发系统的速率限制。

另外，Gemini API 还提供了一个 “AI Studio” 界面，可以在浏览器里直接调试模型参数，这对于不熟悉代码的同学来说是一个很便利的入口。通过 AI Studio，你可以实时看到模型的 token 输出情况，调整 temperature、top_p 等参数，找到最适合自己的生成风格。我在这儿尝试了几次后，发现把 temperature 调到 0.7 附近，top_p 设为 0.9 时，生成的文本既有创意又不至于太随意。对于需要结构化输出的任务，比如抽取实体或者生成 JSON，我发现把 “response_format” 参数设为 “application/json” 可以让模型直接返回符合规范的 JSON，省去后处理的麻烦。

在 OpenRouter 上，我主要用它来测试一些开源的多模态模型，比如 LLaVA、BLIP 等。因为 OpenRouter 提供了统一的 API，我可以用同一套请求代码，分别调用不同模型的接口，这省去了切换 SDK 的麻烦。在实验中，我发现 OpenRouter 对于某些模型的响应时间比 Gemini API 稍微长一点，但因为它的免费 tier 没有并发限制，所以整体吞吐量还是可以接受的。尤其是在做小规模的原型验证时，OpenRouter 的免费 tier 让我可以随意尝试不同的模型组合，而不必担心配额耗尽。

总的来说，Gemini API 的免费 tier 适合那些对调用次数有明确上限需求，并且想要在官方平台上获得相对稳定的服务的开发者。它的优势在于模型质量高、官方文档完善、支持的多模态功能齐全。但它的限制在于配额和并发数量，对于大规模实验可能不够友好。OpenRouter 的免费 tier 则更适合那些想要快速切换模型、进行对比实验的开发者，尤其是在资源受限的情况下，它提供的 1 万次免费调用可以让你先行评估模型的性能，再决定是否付费使用。

在使用过程中，我还遇到一些小问题。比如在 Gemini API 中，如果请求的 payload 超过了 8KB，系统会直接返回 400 错误，这意味着如果你想发送更长的上下文，需要把它拆分成多个请求。我一开始就想一次性把整段文档发给模型进行总结，结果直接报错，后来才改成分段处理。还有在 OpenRouter 上，有时候会出现 “rate limit exceeded” 的提示，这时候只能等待一段时间后再试，或者改用更低频率的请求。这些细节在官方文档里都有说明，但第一次使用时可能会忽略，导致不必要的错误。

如果你想把这些免费资源用到实际项目里，我建议先做好配额监控。可以在每次请求后把请求计数写进本地文件，或者使用云函数定时统计，这样就能随时知道还剩多少免费次数。另外，合理安排请求的时间窗口也是一个关键点。比如在本地开发时，可以把请求集中在深夜或者早早上，避开高峰期，这样成功率更高，响应也更快。还有一点，就是尽量使用缓存。对于相同的输入，如果已经得到过相同的输出，就可以直接复用，而不是每次都重新请求，这样不仅能省掉配额，还能提升整体响应速度。

在安全性方面，Gemini API 提供了 API 密钥的细粒度控制，你可以为不同的项目创建独立的密钥，这样即使密钥泄露也不影响其他服务。OpenRouter 则相对简单，只能通过统一的密钥进行访问，但它也支持在请求头里携带额外的认证信息，以防止滥用。总的来说，两者在安全性上都算是比较可靠的，只要遵守官方的使用规范，一般不会出现安全风险。

最后，我想聊聊对未来的展望。随着大模型技术的快速迭代，免费 tier 的配额可能会有所调整，但大方向应该是继续保持一定的免费使用额度，让更多开发者有机会接触先进的 AI 能力。对我个人而言，我会继续利用好这些免费资源，做一些小实验，然后把结果分享到社区，帮助更多人了解这些模型的实际表现。希望我的这篇随手记录能对正在寻找合适 API 的同学们有一点帮助，如果还有其他细节想了解的，欢迎在评论区留言交流。

【描述】
最近在社区里沸传的 Gemini API 免费额度实测消息，让很多开发者开始关注到底能免费调用多少次。这篇文章把最新的配额数字、使用限制以及实际体验全部拆解，帮助你快速判断是否值得在项目里尝试。

Gemini API 提供的免费 tier 包括每月 5 万次请求，并发上限 10 次，计算资源 1000 小时；而 OpenRouter 的 free tier 则是首 1 万次调用免费，且没有并发限制。两者的配额和使用规则不同，决定了它们适合的实验规模和使用场景。

在实际项目中，我把请求拆成 5000 次一批，间隔 0.2 秒，既不触发速率限制，又能在免费额度内完成全部图像标注。AI Studio 让参数调优变得直观，而 OpenRouter 统一的接口则方便快速切换不同模型进行对比实验。

优势在于模型质量高、文档完善，但配额和并发限制让大规模批处理不太友好；OpenRouter 的免费额度虽大，却在高峰期可能排队，需要合理调度。总体来看，这两个平台互补，能够满足从原型验证到小规模部署的不同需求。

如果你想亲自体验，建议先在本地搭建请求计数脚本，监控剩余配额，避免突发超限。欢迎在评论区分享你的实验结果或遇到的问题，交流使用技巧。
