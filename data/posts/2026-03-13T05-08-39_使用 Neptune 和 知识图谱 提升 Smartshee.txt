【标题】
使用 Neptune 和 知识图谱 提升 Smartsheet 推荐效果

======================================================================
【正文】
大家好，我是 Lucy，来自小红书的大模型观测站。今天想和大家聊聊我最近读到的一篇 AWS 官方博客，讲的是 Smartsheet 怎么用 Amazon Neptune 和知识图谱来做推荐系统的改进。其实一开始看到这个标题的时候我就挺好奇的，Smartsheet 本来是做协作和项目管理的，和传统的电商或内容平台不太一样，为什么也要做推荐呢？带着这个疑问我把文章看了一遍，下面把我的理解和一些细节分享给大家，顺便也说说我觉得哪些地方还有点模糊，或许能给正在做类似事情的同学一些参考。

首先说下 Smartsheet 到底在做什么样的推荐。我们知道 Smartsheet 提供的是一种在线的表格和工作流平台，用户可以在上面建立各种项目、任务、看板等等。随着用户数量和数据量的增长，平台里面积累了大量的表格模板、自动化规则、以及用户之间的协作记录。这些信息其实蕴含着很多隐含的关联，比如某个行业的用户经常会用某种特定的模板，或者某种自动化流程在特定的团队里反复被使用。如果能把这些关联挖掘出来，就可以在用户创建新表格或者设置工作流的时候，给出一些个性化的建议，比如推荐适合的模板、自动化脚本，甚至是可能感兴趣的协作伙伴。这其实就是一个典型的基于图的推荐场景：用户、模板、规则、团队等等实体之间形成了一个复杂的关系网，我们需要在这个网络里找出最相关的节点。

那么为什么选用 Amazon Neptune 呢？文章里提到，Smartsheet 之前尝试过用传统的关系型数据库和一些基于矩阵分解的方法来做相似度计算，但发现随着图的规模扩大，查询延迟和维护成本都开始变得不可接受。特别是当他们想要加入更多类型的关系的时候，比如用户对某个模板的点击、评分、甚至是在评论里提到的关键词，传统的做法就很难灵活扩展。这时候他们看到了 Neptune 这个托管的图数据库，支持 Property Graph 和 RDF 两种模型，而且有开源的查询语言 Gremlin 和 SPARQL。文章里说，他们最初是被 Neptune 的全托管特性吸引的——不需要自己管理集群、备份、版本升级这些运维工作，可以把更多精力放在数据建模和算法上面。不过我也看到他们在迁移的过程中遇到了一些小插曲，比如最初把一些复杂的循环关系建模成了 Property Graph，结果在跑某些深度遍历查询的时候发现性能不如预期，后来调整成了更偏向于 RDF 的方式，用了一些自定义的词典和本体，才让查询更加稳妥。这个过程其实挺典型的，很多团队在选图数据库的时候会被功能列表吸引，但在实际落地时还是要根据查询模式来做相应的建模调整。

接下来是知识图谱的构建。文章里讲到他们把 Smartsheet 平台里的几类核心实体抽象出来：用户(User)、模板(Template)、工作流(Workflow)、自动化规则(Rule)、团队(Team)、以及一些行为事件(Event)比如点击、收藏、分享等。每种实体都有一组属性，比如模板有行业标签、复杂度评分、创建时间等等；用户有所属部门、历史使用频率、偏好标签等等。这些属性在 Neptune 里就是顶点的属性。而边则表示各种关系：用户-使用-模板、模板-属于-行业、工作流-包含-规则、团队-拥有-用户、用户-点击-事件等等。此外他们还引入了一些基于行为的加权边，比如用户在过去30天里对某个模板的点击次数越多，边的权重就越高。文章里提到他们用了一个衰减函数来把原始计数映射到[0,1]之间的得分，具体公式大概是 w = 1 - exp(-k * count)，其中 k 是一个经过实验调参得到的常数。这里我有点好奇他们到底是怎么选的 k 值，文章里只说了是根据线上实验调的，但没有给出具体数值，这点其实是很多工程团队在做特征工程时会遇到的细节，建议大家如果有类似需求可以多做几组对照实验，看看不同的衰减速率对最终推荐效果的影响。

有了图之后，他们接下来要做的就是基于图的相似度计算和推荐生成。文章里说他们主要采用了两种策略：一种是基于元路径的相似度（Meta-path based similarity），另一种是基于图嵌入的方法（Graph Embedding）。元路径的思路其实不复杂，就是预先定义一些在图里面走的路径模式，比如 User -[使用]-> Template -[属于]-> Industry -[属于]-> Template'，这样的话两个用户如果都经常使用同一行业下的模板，就在该元路径下会得到较高的相似度得分。他们在文章里列举了几条他们觉得有用的元路径，比如 User -[点击]-> Event -[关联]-> Template 和 Team -[拥有]-> User -[使用]-> Template。这些路径的选择其实也有一定的经验成分，他们提到一开始尝试了很多路径，后来通过离线实验发现只有不到五条路径对提升命中率有明显贡献，其他的反而会引入噪声。这里我感觉他们其实做了一些特征选择的工作，只不过没有在博客里面详细展开，只是说"经过多次实验筛选"。这也是我想提醒大家的一点，在做图特征的时候，不要贪多，要有明确的假设和验证手段。

另一种方法是图嵌入。文章里说他们尝试过 DeepWalk、Node2Vec 以及最近比较流行的 RGCN（关系图卷积网络）。因为他们的图里边类型比较多（有使用、点击、属于等等），所以他们最终选择了 RGCN，因为它能够在聚合邻居信息的时候考虑边的类型差异。具体来说，他们为每种边类型学习一个不同的权重矩阵，在每一层的聚合步骤里，先把邻居节点按照边类型分组，再分别做线性变换和加权求和。文章里提到他们用了两层 RGCN，嵌入维度设置为 128。训练的时候他们把图看作是无监督的，目标是让连接的节点在嵌入空间里更近，未连接的节点更远，使用的是负采样的方式。这里我有点疑惑他们到底采样了多少比例的负样本，文章里只说"根据经验取了 1:5 的正负比例"，但没有说明是怎么采样的（是随机选取非邻居节点还是根据度做偏采样）。这个细节对复现影响可能不大，但如果想在自己项目里直接借鉴的话，还是建议自己再做一些实验来确定最合适的负采样策略。

说了模型，接下来是怎么把这些相似度或者嵌入转化成实际的推荐列表。文章里说他们在线上服务里做了一个两阶段的流程：第一阶段是候选生成，就是根据用户的当前上下文（比如他正在编辑的表格类型、所属团队等等）在图里做一次有限深度的遍历或者直接查询预计算好的近邻表，拿到几百个候选项；第二阶段是排序，用之前得到的相似度分数或者嵌入的余弦相似度作为特征，再加上一些业务特征比如模板的新鲜度、团队的匹配度等等，喂给一个简单的梯度提升树（GBDT）做最终的排名。这里我觉得挺有意思的，他们把图模型的输出当作特征输入到传统的机器学习模型里，这种做法其实在很多工业界的推荐系统里都很常见，因为图模型往往擅长捕获高阶结构信息，但直接用它的原始得分做排名可能会欠缺一些业务上的细节调节。通过 GBDT 二次加工，可以更灵活地融入一些诸如多样性、新鲜度、甚至是 A/B 测试的干预因子。文章里提到他们在线上实验中看到，相比只用图相似度直接排名，加入 GBDT 后点击率提升了大约 12%。这里的数字我也标注了来源，就是根据 AWS 官方博客里的实验结果部分。

除了算法之外，他们还花了不少精力在数据管线和特征工程上。文章里说他们每天会跑一次 ETL 作业，把 Smartsheet 各个服务产生的日志（点击、编辑、评论等）写入到 Kafka，然后用 Flink 做流式聚合，生成实体的属性更新和边的权重更新，最后写入到 Neptune。这里我特别想提一下他们提到的"延迟容忍度"：因为他们觉得即使图数据有几个小时的延迟也不会对推荐产生太大影响，所以他们把批处理窗口设置成了 4 小时。这个决定其实也是基于他们对用户行为的观察——用户在同一天内重复使用同样的模板的概率不高，而跨天的偏好变化更为缓慢。不过我也觉得这个假设可能并不适用于所有场景，比如如果你的平台有很强的时效性（比如新闻或者热点话题），那就可能需要更实时的更新。这里就是我想说的，架构决策一定要结合具体业务的特性来看，不能 simplesmente 照搬别人的做法。

在效果方面，文章里给出了一些线上实验的数据说明。他们把引入 Neptune 和知识图谱之前的旧方案作为基线（基线主要是基于协同过滤的矩阵分解加上一些规则），然后分三组做实验：A 组只使用元路径相似度，B 组只使用图嵌入（RGCN），C 组是两种特征都用并且加了 GBDT 排序。结果显示，单独使用元路径相似度相比基线提升了 6.3% 的点击率，单独使用图嵌入提升了 8.1%，而组合使用后提升达到了 12.5%。这些数字同样来源于 AWS 官方博客的实验表格部分。我觉得这些提升幅度其实已经很不错了，特别是考虑到 Smartsheet 的用户群体相对比较专业，需求比较明确，能够在这个基础上再拿到双位数的提升说明图的信息确实带来了额外的价值。

不过我也注意到文章里有些地方描述得比较概括，比如他们提到"在某些特殊场景下，模型会出现过度泛化的情况"，但是没有具体说是哪些场景，也没有给出对应的缓解措施。这里我猜可能是当新用户或者新模板刚进来的时候，因为图里连接非常稀少，嵌入容易被拉向一些高度连接的老节点，导致推荐不够个性化。我觉得如果要改进这一点，可以考虑在嵌入阶段加入一些基于属性的相似度做正则化，或者在候选生成阶段强制加入一些新品探索的机制。这其实也是图推荐常见的挑战：如何在利用已有结构的同时，还能给新实体留出展示的空间。

还有就是关于模型更新频率的讨论。文章里说他们目前是一天更新一次嵌入和近邻表，但有在探索更频繁的增量更新方案。他们试过用 Neptune 的流触发器（Stream）来捕获图的变动，然后只更新受影响的节点的嵌入，但发现实现起来复杂度挺高，尤其是当图里有很多高度相连的核心节点时，一次小的边更新可能会导致大范围的重新计算。这里其实也暴露出图嵌入在增量场景下的一个痛点：很多传统的嵌入算法是设计成批量计算的，增量更新往往需要额外的近似或者重新训练某些局部。这也是我个人觉得值得继续研究的方域，如果有同学在做类似事情的话，可以关注一下最近的一些工作，比如增量图神经网络或者基于采样的在线学习方法。

最后想说一下我个人的一点体会。做推荐系统的时候，我们经常会被各种花里胡哨的新模型吸引，但其实能够在业务里落地、并且带来可测量提升的往往是那些把数据管线、特征工程和线上评估做扎实的方案。Smartsheet 这个例子虽然看起来挺技术的，但背后其实是他们在理解用户行为、建立合适的图模型、以及把模型输出与业务规则结合方面做了一系列的细致工作。当然，也不是说他们完全没有问题，文章里也透露出他们还在努力解决新客体冷启动、模型更新时延以及如何在多目标之间做平衡这些问题。不过在我看来，这些恰恰是技术发展的正常状态——没有方案是一劳永逸的，能够不断迭代、根据实际反馈来调整才是关键。

如果大家也在做图相关的推荐或者想用 Neptune 来试一下知识图谱的思路，建议可以从以下几个点入手：首先明确你想要建模的实体和关系种类，别一上就把所有可能的边都塞进去；其次做一遍离线的可解释性分析，看看哪些元路径或者哪种嵌入方式真的对你的目标指标有帮助；最后记得把模型的输出当作特征去喂给后续的排序模型，这样更容易加入业务上的调节。希望我的分享能对大家有所帮助，也欢迎大家在评论区里交流你们在图推荐或者 Neptune 上面的经验和踩坑，毕竟真正的进步还是靠大家一起讨论和碰撞出来的。好了，今天的内容就先聊到这里，我们下次再见！
