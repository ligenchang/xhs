AI 生成代码是否构成衍生作品？清室实现的新挑战
开源再许可争议： Coding Agent 能绕过 LGPL 吗？

选择标题：从 chardet 案例看 Coding Agent 重写开源项目的法律风险

大家好，我是 Lucy，来自小红书的大模型观测站，我们专注于挖掘和分享最新、最有用的AI工具和模型，帮助开发者提升效率，抓住AI浪潮中的机会。

最近一直在看关于 coding agent 能不能用清室实现来把开源项目重新许可的讨论，感觉这个话题真是越来越热了，尤其是在 chardet 这个老牌库上发生的事情，让人不得不停下来想想其中的法律和技术细节。我不打算在这篇文章里给出一个绝对的结论，只是把我看过的材料、看到的争议以及自己的一些想法写出来，供大家参考，如果说有什么地方不太严谨或者写得有点随意，那就是我本来就想像开发者随手写一篇博客那样子，不是为了做学术报告。

首先得说说什么叫清室实现。最经典的例子还是上世纪八十年代 Compaq 克隆 IBM BIOS 的那个故事，他们把一个团队负责反向工程出规格文档，另一个团队完全看不到原始代码，只根据那个规格文档从零开始写实现。这样的话，新写的代码在法律上就不被认为是原始代码的衍生作品，因为两团队之间没有直接的代码共享。这个过程原来要花好几周甚至几个月，现在有了 coding agent，好像只要几个小时就能搞出一个类似的“清室”产物——虽然它其实并不是真正的清室，因为人还是得先看过原始代码才能给出提示或者设计文档。

接下来就是 chardet 这件事了。chardet 最早是 Mark Pilgrim 在 2006 年写的，后来由 Dan Blanchard 长期维护，最近 Dan 把 7.0.0 版本重写了一遍，声明是完全重构，MIT 许可证，包名和 API 没变，宣称更快更准确。就在发布不久后，Mark Pilgrim 在 issue #327 里提出质疑，说 Dan 没有权力把许可证从 LGPL 改成 MIT，因为即便是重写，也受到原始代码的影响，而且 Dan 本人维护了这么多年，显然对原来的代码非常熟悉，这显然不符合传统清室的要求——也就是得有两个互不知情的团队才行。

Dan 的回复其实也挺坦率的，他承认自己确实对旧代码有很深的了解，不过他觉得清室的目的不是为了形成一种流程上的隔离，而是为了确保最终得到的代码在结构上不依赖旧代码。为了证明这一点，他拿出了 JPlag 检测工具的结果，说新旧版本之间的最大相似度只有 1.29%，和早期的 1.1 版本比的话相似度只有 0.64%，而之前的几个版本之间的相似度却在 80% 左右浮动。他认为这么低的相似度足以说明新代码在结构上已经基本独立了。

不过这里还有一些细节值得推敲。Dan 说他在重写的时候用了什么“超级大脑 storming 技能”来先写设计文档，然后在一个完全空的仓库里让 Claude 生成代码，并且特别告知 Claude 不要去参考任何 LGPL 或 GPL 许可的代码。他还说他自己会逐块审查、测试、迭代，确保生成的东西符合他的预期。这个过程听起来挺像是让 AI 当成了一种辅助设计工具，但与此同时，他自己作为维护者的知识背景显然还是绕不过去的。

从法律角度看，LGPL 明确要求基于它的改动也必须继续采用 LGPL，除非你能证明你的作品不是衍生作品。传统清室正是用来做这个证明的——通过隔离团队来确保没有直接复制。可是一旦维护者自己就是原来代码的熟悉者，这种隔离就不存在了。即使新代码在文本相似度上很低，法律上仍然可能会被认为是受到原始代码的影响，特别是当你明确知道自己曾经读过、修改过那些代码的时候。这其实有点像在写小说时虽然你没直接抄袭别人的句子，但如果你深受某种风格影响，仍然可能被认定为衍生。

再说说技术层面。 coding agent 确实能够在短时间内生成大量代码，而且可以通过 prompt 控制它不要去看某些特定许可证的代码。可是问题是，这些模型本身是在海量公开代码上训练的，其中肯定包括大量 LGPL、GPL 等等开源项目的代码。即便你告诉它“不要基于 LGPL 代码”，它内部的权重其实已经吸收了那些代码的模式和结构。所以从某种意义上说，使用这样的生成模型本身就带有一种“间接的接触”。这一点在学术界和法律界还是一个灰色地带，目前还没有明确的判例来说明这种情况是否算作衍生。

回到 chardet 案例，Dan 给出的 JPlag 数字看起来很有说服力，可是我们也要知道相似度低并不等于没有任何关联。比如两段实现同样功能的代码，如果它们采用了完全不同的算法或者数据结构，相似度自然会很低，这时候即使有代码共享也可能被稀释掉。反过来，如果两段代码虽然相似度高，但其实都是一些通用的样板代码（比如常见的错误处理、日志打印），也不一定构成侵权。所以单纯依赖一个百分比来判断是否构成衍生作品可能还是不够全面。

在这件事情上，我个人倾向于认为： coding agent 可以当作一种高效的原型工具，帮助开发者快速生成想法或者探索不同的实现路线，但在涉及到改变开源项目许可证这种敏感操作时，还是应该格外小心。如果项目的维护者本人对旧代码有深度了解，那就很难主张自己进行了传统意义上的清室实现。即便使用 AI 生成的代码在文本相似度上表现很好，法律上的风险仍然存在，更何况现在围绕 AI 生成代码的版权归属问题本身还在激烈讨论中。

当然，我也能理解 Dan 的动机。重写一个十年的老项目，为了提升性能、修复漏洞、现代化代码基础，确实是一件值得鼓励的事情。而且他也确实付出了大量的时间和精力去做这件事，不是简单地丢给 AI 就完事了。只是在许可证这一块，也许可以考虑其他的途径，比如继续保持 LGPL，或者在得到原始版权持有人（虽然 Mark 已经淡出）的明确许可后再尝试更宽松的许可。毕竟开源社区的健康离不开对规则的尊重，也离不开贡献者的努力。

最后想说的是，技术发展总是会把法律和伦理推向新的边界。 coding agent 能力的提升固然让我们看到了以前不可能实现的效率，但与此同时，我们也得花时间去思考怎样用这些新工具，既不失去创造的自由，也不忽视开源共同体所依赖的那些共享规则。如果大家有实际操作经验或者法律上的见解，欢迎在评论区分享，互相学习才是我们进步的路子。

【描述】
最近，关于 Coding Agent 能否通过清室实现把开源项目重新许可的讨论愈演愈烈，尤其是 chardet 库从 LGPL 改为 MIT 的争议，让不少开发者开始审视 AI 生成代码的法律边界。这不仅关系到许可证合规，还可能影响未来开源项目的维护策略。

清室实现的核心是两个互不知情的团队：一方只输出规格文档，另一方完全凭文档从零编码，以确保产出不被视为原始代码的衍生作品。在 chardet 案例中，维护者 Dan 本人对旧代码有深度了解，尽管他使用 Claude 生成新代码并在 JPlag 检测中得到极低的相似度（约1%），但法律上仍可能因其“间接接触”而被认定为受原始代码影响。此外，AI 模型本身是在包含大量 LGPL/GPL 代码的语料上训练的，即便 prompt 指令禁止引用，其内部权重也已吸收相应模式，这使得“真正的清室”变得更加难以证明。

对于开发者而言，Coding Agent 仍是一个高效的原型工具：可以快速生成功能雏形、探索不同算法路径，甚至在全新项目中辅助编写样板代码。但在涉及许可证变更的敏感场景，建议采取以下措施：一是让未接触过原始代码的团队负责规格制定；二是由完全不同的团队或 AI 在空白仓库中实现；三是完成后进行专业的法律合规审计，特别是检查是否存在潜在的衍生作品风险。这样既能保留 AI 带来的效率提升，又能降低许可证纠纷的可能性。

从深度观点来看，单纯依赖文本相似度低来判断独立性是不够的；功能等价、结构相似乃至开发思路的延续都可能构成法律上的衍生。AI 生成代码的“黑箱”特性让传统清室的隔离假设受到挑战，未来可能需要新的法律框构或行业指南来界定 AI 辅助开发的衍生作品标准。我个人认为， Coding Agent 更适合用于加速创新而非直接规避开源协议，尊重原始许可证仍是维持开源生态健康的基石。

如果你正在考虑用 Coding Agent 重构或重新许可项目，不妨先在非协议限制的小型尝试中试水，观察生成代码的相似度和功能表现，并在评论区分享你的做法和疑惑。期待看到大家的实践经验和法律见解，一起在技术与规则之间找到更平衡的路径。
