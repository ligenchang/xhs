【标题】
SWE‑Bench 终结：OpenAI 为何放弃并推出 Pro

======================================================================
【正文】
OpenAI 最近在内部评估中正式宣布，SWE‑Bench Verified 将不再作为公开的基准使用，转而聚焦于更具挑战性的 SWE‑Bench Pro。这一决定并非突发，而是经过数月的内部讨论和数据验证后才公开的。我们可以从几个关键角度来拆解背后的原因和意义。

首先，SWE‑Bench 的最初设想是为了给大模型在实际软件工程任务上的表现提供一个可比的参考。它收录了 500 条经过筛选的 GitHub Issue，要求模型在不依赖外部知识的前提下，仅凭代码生成能力完成任务。早期的实验显示， frontier model 在通过 SWE‑Bench Verified 时，往往能够达到 80% 以上的通过率，甚至在某些模型上出现轻微下滑的情况。于是社区开始调侃，每次发布新模型时，分数只会微调几位小数，像是 0.1% 的微调，显得异常稳定。

然而，OpenAI 在内部对 138 个被标记为 “有问题” 的任务进行了更深入的审查。他们邀请了至少六位工程师对每个问题进行二次复核，发现有 60% 的剩余任务在严格意义上是不可解的。更具体地说，这些任务要么是过度狭窄 —  — 只接受特定的实现方式，而排除掉了功能上等价但写法不同的代码；要么是过度宽泛 —  — 要求模型额外实现文档中未提及的功能。这样一来，模型在实际使用中往往需要“偷懒”，只针对这些特殊需求进行训练，才能拿下分数。

其次，OpenAI 发现了一个更加致命的问题 —  — 训练数据的泄露。SWE‑Bench 的任务来源于公开的仓库，而这些仓库本身也是许多模型训练时使用的材料。随着 SWE‑Bench 的流行，问题描述和对应的金色补丁（gold patch）逐渐渗透到模型的训练语料库中。于是，模型只需要记住一个 Task ID，就能在极短的提示下复写出原始问题的完整答案，甚至直接复制金色补丁。换句话说，很多所谓的 “通过” 其实是模型已经在背后记忆了答案，而不是真正展示出了推理或实现能力。

这种现象在所有 frontier model 中都有所体现，尤其是 OpenAI 自身的模型在面对仅凭 Task ID 的提示时，就能毫不费力地输出与原始问题完全相同的解决方案。人工审查员很快就能识别出这些 “不可能的题目”，因为它们的解决方案往往与原始问题的描述毫无关联，却又能精确匹配到代码片段。换句话说，这些题目已经失去了真正的评估意义，成为了模型炫耀的秀场。

基于以上发现，OpenAI 决定在公开场合正式终止 SWE‑Bench Verified 的报告，并推出全新的 SWE‑Bench Pro。Pro 版本的核心目标是转向更开放式的任务，强调长期任务的连贯性、开放式设计决策、代码质量以及可维护性。换句话说，未来的评估不再是单纯的 “能否通过一道题”，而是要看模型在真实项目中能否持续产出高质量、可维护的代码，能否在多步骤的任务链中保持逻辑一致性，还能在面对模糊需求时给出合理的设计方案。

在技术细节上，SWE‑Bench Pro 将采用更宽松的评估标准，允许模型在解决问题时展示完整的链式思考（chain‑of‑thought），而不是仅仅给出最终的代码片段。评估者会对代码的可读性、注释的完善性、模块化程度以及是否遵循工程最佳实践进行打分。此外，任务本身会设计成需要跨多个文件进行交互的情形，迫使模型在实际项目中进行文件结构的组织和接口的定义。这样一来，模型的表现更加接近于真实的软件开发场景，而不是单纯的填空题。

社区对这一变动的反应是复杂的。一方面，许多研究者认为这是一次必要的进步，因为旧有的基准已经被“饱和”，不再能反映前沿模型的真实能力。另一方面，也有不少开发者担心新基准的难度设置会不会过高，导致评估结果过于苛刻，甚至可能因为实现细节的差异而导致不公平的比较。更值得关注的是，OpenAI 在公开声明中提到，他们正在借鉴自己在 GDPVal 项目中使用的 rubric‑based 评估方式，即通过多维度的评分矩阵来量化模型的表现。这样一来，评估不再是简单的通过/失败，而是一套细致的指标体系。

总的来说，SWE‑Bench 的终结标志着人工智能评估领域的一次重要转折。它提醒我们，单一的、高度抽象的基准往往无法捕捉模型在真实世界中的综合能力。只有在更宽松、更具开放性的任务框架下，才能真正检验模型的长期推理、代码质量和工程实践。对我们这些日常与代码打交道的开发者而言，这也意味着未来在选型和部署模型时，需要更加关注模型在实际项目中的表现，而不仅仅是某个固定榜单上的分数。

在实际操作层面，如果你想在自己的项目中使用 SWE‑Bench Pro，首先需要关注以下几个方面：一是任务的描述是否足够开放，能否让模型在不依赖预设答案的情况下展开思考；二是评估 rubric 是否覆盖了代码可维护性、设计决策的合理性等关键维度；三是是否有足够的样本数据用于对模型的输出进行统计分析，避免单次实验的波动影响结论。只有在上述环节都做好，才能确保评估结果的可靠性和可复制性。

最后，值得一提的是，OpenAI 并没有把所有精力都放在 SWE‑Bench Pro 上，他们仍然在推进其他更具挑战性的评估项目，比如围绕实际产品构建的端到端任务、跨模态交互的复杂工作流等。可以预见，未来的 AI 基准体系将会更加多元化，也更加贴近真实工程需求。对于我们这些在产品研发中不断尝试新模型的开发者来说，保持对这些新基准的关注，并及时调整自己的评估方法，是保持技术前沿的关键一步。

【描述】
OpenAI 刚刚公告将停用 SWE‑Bench Verified，转而推出更具挑战性的 SWE‑Bench Pro。这背后不只是一次基准切换，而是开发者评估模型方式的重大转折，值得我们第一时间关注。

1️⃣ 过去的 SWE‑Bench Verified 通过率高但停滞，60% 标记为不可解的任务暴露了模型的“记忆”依赖；2️⃣ 训练数据泄露导致模型直接复写金色补丁；3️⃣ 新版 Pro 强调开放式、跨文件任务，聚焦代码质量、可维护性和长期链式推理。

使用 SWE‑Bench Pro，模型能够展示完整的链式思考、代码可读性和模块化设计，而不是单纯填空。它特别适合需要持续交付可维护代码的团队、原型研发以及自动化 CI 流程。

亮点在于把评估维度从“能否通过”扩展到可维护性、设计决策和跨文件协作；局限是任务难度可能过高，导致不同模型的可比性下降。整体趋势指向更贴近真实工程的全局评估，这正是行业需要的进化。
