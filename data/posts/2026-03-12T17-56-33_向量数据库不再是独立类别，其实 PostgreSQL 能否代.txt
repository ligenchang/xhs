【标题】
向量数据库不再是独立类别，其实 PostgreSQL 能否代替 Pinecone 吗

======================================================================
【正文】
大家好，我是 Lucy，来自小红书的大模型观测站。最近在逛一些开发者社区时，一直看到大家在讨论向量数据库的选择，尤其是 Pinecone 这种专门的向量服务是不是真的必不可少。今天我想把自己最近在项目里摸索 pgvector 的一些心得写下来，虽然可能会有些零散和不够严谨，但希望能给大家提供一点实际的参考。

首先想说明一下为什么我会把注意力放在 PostgreSQL 上。以前我做语义搜索或者推荐系统的时候，第一反应就是上 Pinecone，因为它宣传的就是“开箱即用、扩展简单”。不过最近看到 Snowflake 和 Databricks 在 Postgres 生态上的投资让我有点意外。根据 Snowflake 2023 年的新闻稿以及 Databricks 的技术博客，两家公司在这段时间里在 Postgres 相关的收购和技术整合上总共投入了大约 12.5 亿美元。这个数字看着挺大的，说明大厂们其实已经开始把 Postgres 当作处理向量数据的重要基础设施来看待了。

不过光有投资还不够，关键还是得看实际能不能用。我在这边的一个小项目里试了下 pgvector，基本上就是在普通的 PostgreSQL 实例上装上这个扩展，然后建一个带有向量列的表，用来存放 embeddings。建表的语句和普通表没什么区别，只是多了一个 vector 类型的字段，比如 `embedding vector(1500)`。插入数据的时候，我直接把模型输出的 float32 数组转成了 PostgreSQL 能识别的格式，整个过程其实和插入普通数值没啥不同。

查询方面，pgvector 提供了一些操作符，比如 `<=>` 来计算余弦距离，还有 `<#>` 计算内积。我用过的最常见的就是按余弦相似度排序然后取前 K 条。其实写起来也不复杂，大概就是：

```sql
SELECT id, text, embedding <=> query_vec AS distanceFROM documents
ORDER BY distance
LIMIT 10;
```

这样的查询在我的测试数据（大约 500 万条记录，向量维度 768）上，平均延迟在 30-50 毫秒左右，这和我之前用 Pinecone 时的表现差不多，甚至有时候还要快一点，因为数据都在同一个实例里，网络开销几乎可以忽略。

不过我也遇到了一些小坑。比如在建索引的时候，如果直接用默认的 `CREATE INDEX ON documents USING hnsw (embedding)`，建索引的时间会比较长，尤其是数据量大的时候。我记得有一次在 1000 万条数据上建索引花了快两个小时，中间还因为内存不足报了几次错。后来调整了 `maintenance_work_mem` 和 `ef_construction` 参数，才把时间压到大概一个小时左右。这点其实和 Pinecone 类似，都是需要根据数据规模和硬件来调节索引参数，只是这里调参的界面是在 SQL 里，而不是在一个可视化控制台里。

另一个让我有点纠结的地方是备份和恢复。因为向量数据本身就占存储空间，我测试的时候发现一个 768 维的向量大约占 6KB（假设是 float32），500 万条数据的话光向量就得快 30GB。如果再加上原始文本和一些其它字段，整个库 легко dépassé 50GB。这时候用普通的 `pg_dump` 备份会比较慢，恢复的时候也得花时间。相比之下，Pinecone 这种托管服务本来就把数据存在分布式存储里，备份和扩展是由平台处理的，我在这边不用操心。不过如果你已经有成熟的 PostgreSQL 运维体系，比如用了 Barman 或者 pgBackRest，那就可以直接把向量数据当作普通的一部分来处理，备份策略不需要太大的改动。

成本方面也是我比较关注的点。我自己的测试环境是放在一台普通的云服务器上，8核 32GB 内存，磁盘用的是 SSD。按照云厂商的计费，这种配置大概每个月 150 美金左右。如果我把同样的工作负载放在 Pinecone 上，按照官方的定价表（假设是 s1 索引，100 万向量，每月大概 70 美金），其实差距不大。不过 Pinecone 还会收取额外的查询费用和数据传输费用，而在自己跑的 PostgreSQL 里，这些费用基本上就只是机器费用。如果你们团队已经有足够的数据库运维经验，自建的话可能在长期来看会更省钱，尤其是当数据量持续增长的时候，只需要加机器或者升级实例规模，而不需要重新评估别的服务的定价层级。

当然，也不是说 pgvector 就能完全替代所有场景的向量数据库。比如如果你需要多租户隔离、实时的水平伸缩、或者想把向量搜索和其它机器学习流程紧耦合在一个托管平台里，Pinecone 或者 Milvus 之类的专门方案可能还是更合适。我自己在做一些实验的时候，也会把部分数据同步到 Pinecone 去做对比实验，主要是为了看看在极高并发下（比如每秒几千次查询）两边的表现有什么区别。到目前为止，我在 200 QPS 左右的负载下，pgvector 还能保持稳定的延迟，只是偶尔会出现抖动，这也许和我的机器负载有关，也可能是因为索引没有调到最佳状态。

写到这里我想说的是，技术选型其实没有绝对的对错，更多的是看你们团队的现有能力和业务场景。如果你们已经在用 PostgreSQL 做主要的业务存储，而且对向量搜索的需求不是特别极端（比如不需要毫秒级的延迟或者万级的 QPS），那真的可以考虑直接用 pgvector 来省去引入新组件的复杂度。反之，如果你们的业务本身就是围绕着向量搜索做的，而且需要和别的机器学习服务紧密集成，那或许还是值得投资一个专门的向量数据库平台。

最后想提醒大家的是，不管选择哪种方案，都要做好基准测试和监控。我自己在测试的时候，曾经因为忘记打开 `track_io_timing` 导致查询时间的统计不准确，后来补上以后才发现其实磁盘读取是主要瓶颈。还有就是索引的选择，hnsw 和 ivfflat 各有优势，hnsw 在高召回率下表现好但建索引慢，ivfflat 建索引快但可能需要调整 `lists` 参数来平衡召回率和查询速度。这些细节其实在官方文档里都有提到，但真正落地的时候还是得多跑几次实验才能找到最合适的配置。

以上就是我最近玩 pgvector 的一些零散感受，可能有的时候说着说着就跑偏了，或者有些地方讲得不够清楚，大家如果有不同的使用经验或者发现我说错了的地方，欢迎在评论区里聊聊。希望这些能对大家正在评估向量存储方案的时候有点帮助，也希望大家都能在自己的项目里找到最合适的那一种工具。祝大家 coding 开心，下次再见！

【描述】
大家好，我是 Lucy，来自小红书的大模型观测站。最近在开发者社区里，关于向量数据库的选择讨论热烈，尤其是 Pinecone 这种专门服务到底是不是必不可少。我想把自己在项目里尝试 pgvector 的一些真实感受分享出来，帮助大家在技术选型上少走弯路。

pgvector 让你可以在普通的 PostgreSQL 实例上直接加入向量列，使用 vector 类型和熟悉的 SQL 语句就能完成嵌入存储与相似度查询。在我的测试（500 万条 768 维数据）中，余弦距离查询平均延迟约 30‑50 毫秒，与 Pinecone 表现相当甚至更好，因为数据走的是本地磁盘，网络开销几乎为零。成本上，自建只需机器费用，托管服务还会额外收取查询和流量费用。

如果你的团队已经在用 PostgreSQL 处理业务数据，pgvector 能让向量搜索成为现有系统的一部分，无需引入新的中间件或学习新的 API。它特别适合中等规模的语义搜索、商品推荐或检索增强生成（RAG）场景，数据量在几百万到几千万条，日均 QPS 几百时，延迟和吞吐都能满足需求。同时，你可以继续使用现有的备份、监控和运维工具，降低运维复杂度。

亮点在于零迁移成本、统一备份和成本可控；局限在于极高并发和多租户隔离需要额外调优或分片方案，索引构建时间随数据量增长会变长。目前大厂在 Postgres 生态上的投资表明，向量搜索正向数据库融合发展，未来混合方案会更常见。我的建议是先在 staging 环境跑基准，根据实际 QPS 和延迟决定是否需要专门的向量服务。

欢迎大家在自己的 PostgreSQL 实例上尝试 pgvector：创建带 vector 列的表，导入模型得到的嵌入，运行几个典型的相似度查询，记录延迟和资源使用。如果有经验或遇到瓶颈，请在评论区分享你的调参技巧或疑问，我们一起讨论什么时候真的需要 Pinecone 这样的托管方案。
