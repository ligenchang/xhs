【标题】
从零到炼厂游戏：Vanilla JS与LLM协同开发笔记

======================================================================
【正文】
大家好，我是 Lucy，来自小红书的大模型观测站。最近我在做一个很有意思的小项目，一个用纯原生 JavaScript 写的炼油厂模拟游戏，主要是为了给孩子们解释我在得克萨斯州炼油厂做的物流工作。本来只是想做个五分钟的小演示，结果一不小心做到了九千行单页应用，期间踩了不少坑，也收获了一些挺实用的经验，想和大家分享一下。

项目的起源其实挺简单的。我每次向家人解释炼油厂的下游操作时，总是遇到一脸懵的表情。孩子们对“电脱盐”、“催化裂化”这样的词完全没有概念，即使我用图片和视频也很难让他们产生兴趣。于是我想做一个交互式的东西，让他们自己动手操作，感受一下原油从地下被抽出来到最终变成汽油、柴油、喷气燃料的全过程。最初的想法只是在网页上放几个按钮，点一下就切换到下一步的静态图片，后来发现这样太枯燥，决定加入一些小游戏来强化概念。

为了实现这些小游戏，我选择了 Matter.js 这个二度物理引擎。它本来是用来模拟刚体碰撞的，我在项目里用它来模拟油滴在分馏塔里的分离、催化剂在裂化装置中的移动等等。一开始我以为只要把物体放进去，让引擎自己处理碰撞和重力就行了，结果发现 Matter.js 对 DOM 元素的边界并不知道。我想让每个小游戏都放在一个圆形的容器里（border‑radius: 50%），这样看起来更像一个个小实验槽。但 Matter.js 的坐标系是基于画布的，它根本不关心你的 CSS。为了让物体在圆形容器里不越界，我不得不在每一帧都检测点到椭圆中心的距离，使用公式 (dx*dx)/(rx*rx)+(dy*dy)/(ry*ry)>1 来判断是否超出边界。虽然这样可以工作，但我在手机和 PC 上的表现不太一样，感觉有点过度工程了。

更让人头疼的是移动端的事件处理。iOS Safari 默认会对双击进行放大，还会让页面整体滚动，这直接干扰了 Matter.js 对拖拽的捕获。我不得不在每个可交互的元素上加上 touch‑action: manipulation; user‑select: none; 还要额外阻止默认的 touchstart、touchmove 事件。奇怪的是，这些设置在 PC 上有时候会把鼠标滚轮事件也给吞掉，导致用户在浏览说明时滚不动。我花了好几天才调到一个似乎还能接受的平衡点，不过仍然会偶尔看到有人反馈在某些老旧安卓机型上触摸不灵敏。

状态管理也是一个大挑战。因为我没有使用 React、Vue 或任何前端框架，所有的数据都放在一个全局的 state 对象里。游戏里有五个主要阶段，每个阶段切换的时候会载入不同的小游戏和对应的 UI。最初我只是在每个阶段开始时创建新的 Matter.js 引擎实例和一些 setInterval 循环用来更新 UI，结果在反复切换之后，内存里开始堆积大量没被清理的定时器和物体对象。页面会变得越来越卡，最终甚至触发浏览器的内存警告。我不得不为每个阶段写一个明确的 teardown 函数，里面把所有的 setInterval 用 clearInterval 清除，把 Matter.js 的 World 清空，再把相关的 DOM 节点从文档中移除。这样虽然有点繁琐，但至少让内存占用保持在一个可以接受的范围。

在实际的炼油流程模拟上，我试图保持一定的科学严谨性，同时又不至于让孩子们看起来像在读教科书。游戏里依次演示了电脱盐（用点击来“击碎”盐水滴）、常压分馏塔（根据不同的沸点将原油分离成 LPG、ナフサ、喷气燃料、柴油和残渣）、加氢脱硫（通过点击硫氧化物来模拟硫的移除）、催化裂化（需要小心管理焦炭积碳和催化剂再生）以及最后的汽油调和（需要达到特定的辛烷值和蒸汽压指标）。每个环节我都加入了一些小提示和反馈，比如在脱盐环节如果误点到油滴会扣分，鼓励孩子们只去处理水和盐的混合物。

整个项目的代码量大约有九千行，全是纯 HTML、CSS 和 JavaScript。虽然我觉得这个数量对一个单页应用来说有点多，但很多都是因为我在处理跨平台事件、物理边界和状态清理时写的样板代码。起初我非常依赖大语言模型（比如 Claude、Copilot、Gemini）来帮我写代码。刚开始的时候，我会让模型直接输出整个 script.js 文件的替换内容，结果经常出现截断、幻觉，甚至把某些直角引号悄悄换成了中文引号，导致 JavaScript 语法错误。后来我改变策略，只让模型给出类似补丁文件的描述——告诉它“找到这段代码”以及“替换成这段代码”。这样虽然每次都要手动复制粘贴，但至少能确保我不引入无法预料的错误，也让我在审查修改时更有掌控感。

除了技术层面的东西，我还学到了很多关于如何把专业知识转化为互动体验的思考。比如在设计分馏塔的小游戏时，我最初想让用户直接拖动油滴到不同的馏程格子里，但这样操作太抽象，孩子们不知道到底在干什么。后来我改成了让用户点击加热炉来升温，然后观察不同颜色的油滴自动上升或下降，再配上简单的文字解释（“轻烃在上方聚集，重油留在底部”）。这样虽然看起来不够“真实”，但至少让玩家能够看到因果关系。

在开发过程中我也遇到了一些意外的问题。比如在测试时发现，某些安卓机型的 Chrome 在连续触发快速点击时会把触摸事件误认为是滚动，导致 Matter.js 的拖拽被意外取消。我后来在文档里看到，这其实是某些浏览器在尝试做“点击延迟”优化时的副产品。我不得不在触摸开始的时候记录时间戳，如果在 150 毫秒内检测到手指移动超过一定阈值，就认为是滚动而不是点击。这样的启发式办法虽然不是百分百保证，但在实际使用里把误判率降到了可以接受的程度。

目前这个游戏已经上线，地址是 fuelingcuriosity.com/game，完全免费，没有广告也不需要注册。我欢迎大家去玩玩看，尤其是如果你对炼油工艺有兴趣，或者恰好在学习前端物理引擎的应用。如果你发现了什么明显的物理 bug（比如油滴竟然可以穿过分馏塔的壁），或者对某些交互的设计有建议，请随时告诉我。我也很乐意回答关于炼油厂实际操作的任何问题，毕竟我在现场待了好几年，了解一些只能从一线得到的细节。

最后想说的是，虽然这个项目一开始只是给孩子们看的科普小工具，但在做的过程中它自己也成了一个很好的前端练手项目。它让我重新审视了如何在没有框架的情况下管理复杂的状态，如何在物理引擎和网页布局之间找到平衡，以及如何在依赖 AI 辅助的同时保持代码的可维护性。希望这篇文字对正在尝试用 Web 技术做科学教育或工业模拟的开发者有些帮助，也希望大家在玩的时候能玩得开心，学到一点东西。祝大家编码愉快，下次见！

【描述】
最近看到 Lucy 用纯原生 JavaScript 做了一个炼油厂模拟游戏，代码接近九千行，却完全不用任何前端框架。这不仅是一个科普小工具，更是一次在浏览器里用物理引擎还原真实工业流程的尝试。对于想了解 Web 如何承载复杂交互的开发者来说，这篇经验分享很有参考价值。

核心经验包括三点：一是用 Matter.js 模拟油滴运动时，为适配 CSS 圆形容器，每帧采用椭圆方程 (dx²/rx²+dy²/ry²>1) 判断越界；二是移动端触摸冲突，通过 touch‑action: manipulation、阻止默认事件以及 150 ms 时间戳阈值区分点击与滚动；三是全局状态下的内存泄漏问题，为每个阶段编写 teardown 函数，清除 setInterval、销毁 Matter.js World 并移除 DOM，才让页面保持流畅。这些解决方案展示了在无框架环境下如何兼顾物理引擎、事件处理和状态管理。

如果你需要在网页上实现带有真实感的物理模拟（比如流体分离、颗粒运动），但又不想引入重量级框架，这个案例提供了可直接借鉴的模式：使用 Matter.js 处理刚体碰撞，配合自定义边界检测来适配任意容器形状；通过统一的 touch‑action 与时间戳启发式方案解决移动端触摸与页面滚动的冲突；采用阶段性 teardown 函数保证状态干净、内存可控。适合科普互动、工业流程演示或任何需要轻量级、高帧率物理交互的场景。开发者可以直接访问 fuelingcuriosity.com/game 查看效果，或在源码中寻找对应的事件处理和状态清理函数进行迁移。
