【正文】
基于 Cloud Run 的多模态 AI 虚拟 staging 方案  如何用 Gemini API 和 Cloud Run 做出 AI 家具布置  
从空房到成图：Gemini + Cloud Run 全流程实录  
多模态 AI 虚拟 staging 实战：Gemini 与 Cloud Run 的结合  

最终选用：用 Gemini 3.1 Flash 实现房间虚拟家具 staging  

大家好，我是 Lucy，来自小红书的大模型观测站，我们专注于挖掘和分享最新、最有用的AI工具和模型，帮助开发者提升效率，抓住AI浪潮中的机会。  

最近我在参加 Gemini Live Agent Challenge 时，想做一个能够把空房照片变成带家具效果图的小工具。本来觉得只要调用一下 Gemini 的图像模型就能搞定，结果发现实际操作里有不少坑。下面我把整个开发过程的思路和遇到的问题说一下，希望对同样想玩多模态 AI 的同学有点帮助。  

最开始的想法很简单：把用户上传的房间照片直接喂给 gemini-3.1-flash-image-preview，让它一次性输出既有家具摆放的图片，又带上营销文案和语音脚本。我当时觉得这样最省事，毕竟只要一次请求就能拿到所有结果。不过实际跑起来的时候，后端服务器经常卡住，前端页面一直显示加载中，最后甚至出现了超时错误。查了下日志才发现，图像模型其实只会返回像素数据，它根本不会理会你让它同时写文字的要求。于是我的代码在等待一个永远不会到来的文本响应时，一直占用着连接，导致 Cloud Run 实例被迫重启。  

这次的教训是：多模态不等于所有模型都能做所有事情。图像模型专注于生成像素，文本模型才擅长理解和生成语言。于是我把后端拆成了三个独立的步骤，每一步只调用最适合的模型。第一步是处理图片，我用 sharp 把上传的原始照片压缩到合适的大小，这样既能保证细节，又能降低传输时间。然后把压缩后的 base64 数据发给 gemini-3.1-flash-image，伴随着一个很明确的视觉提示词，只让它负责在房间里添加家具、调整光影等等。为了防止偶尔的网络波动导致请求挂掉，我给这个 API 调用包装了一个带有重试机制的六十秒超时函数，如果第一次失败就再试一次，最多不超过两次，这样基本能够应对偶尔的突发延迟。  等到图片生成成功以后，我立刻触发第二步：调用 gemini-2.5-flash 生成房产描述和风水建议。这里其实也是多模态的，2.5 Flash 能够同时接受图像和文本输入，所以我把刚才生成的 staged 图片的 base64 再次传过去，并给它一个专门的文案提示词。这一次模型不仅能够看到家具的摆放位置，还能根据房间的布局给出一些风水上的小建议，比如沙发背后最好有靠背，或者灯光的色温要暖一些之类的。生成的文字我会直接保存下来，后面还会用来做语音合成。  

最后一步是把得到的文案送到 Google Cloud Text-to-Speech 去生成音频。我选择了 WaveNet 的英文女声，因为听起来比较自然，而且可以通过调节语速和音高来匹配不同的房产风格。TTS API 返回的是一个 MP3 的二进制流，我直接把它放进内存里，然后连同之前生成的图片和文案一起打包成 JSON 响应发回前端。前端使用 React 搭配 Vite 做构建，Tailwind 负责样式，Framer Motion 负责一些轻微的动画效果，这样用户在看到效果图的时候还能有种逐渐展开的感觉。  

在数据持久化方面，我选用了 Supabase 来做简单的潜在客户信息收集和去重。每次有人上传房间照片并生成结果的时候，我会把匿名的访问 ID、生成时间以及是否点了下一步的按钮记录进去。虽然这只是一个小 demo，但以后如果要把它变成正式产品，这些数据就能够帮助我们分析哪种房型更容易被用户接受，哪种文案更能促成转化。  

整个项目最大的挑战其实在于如何在 Node.js 里优雅地编排这些异步调用。一开始我试图用 async/await 一口气把三个步骤串起来，结果一旦某个环节出错，整条链条就会断掉，而且错误信息也不够明确。后来我改成了使用 Promise.allSettled 来分别处理每一步的成功或失败，这样即使其中一个模型暂时不可用，也能够把已经得到的结果返回给用户，同时在后台记录下错误日志供后续排查。  说到实际效果，我用了一些真实的房源照片做了测试。生成的家具摆放看起来比较自然，沙发和茶几的尺寸基本符合常见的户型比例，光影处理也有一定的层次感。文案方面，模型能够根据房间的朝向给出不同的描述，比如朝南的客厅会强调采光好，朝北的则会提到可以用暖色灯光来营造温馨感。风水建议虽然只是些通用的原则，但也算是给用户多了一个参考维度。语音合成的音质在普通的网络环境下听起来没有明显的机械感，能够直接用于短视频的配音或者线上看房的讲解。  

如果你也想尝试类似的东西，我觉得有几点值得注意。第一，一定要明确每个模型的能力边界，不要期望单一模型能够完成所有任务。第二，在云函数或者 Cloud Run 这种按需实例的环境里，超时和重试机制是必须的，否则一次偶发的网络抖动就可能导致整个服务不可用。第三，尽量把产生的中间结果（比如压缩后的图片、生成的文本）暂存下来，这样在后续步骤出错的时候可以避免从头再来，既节省时间也省下了计算成本。第四，前端的展示可以适当加一点过渡动画，用户会感觉更流畅，也能够在等待结果的时候不至于感到无聊。  

最后说一下我自己在这次挑战中的一点小失误。一开始我太看重一次性完成所有功能的“酷”点，结果花了好几个小时在调试一个根本不可能成功的方案上。后来才明白，有时候把问题拆解得更简单、更明确，反而能够更快地看到进展。希望大家在玩多模态 AI 时也能记住这一点，别被表演效果迷惑了脚步。  好了，今天的分享就到这里。如果你对 Gemini API、Cloud Run 或者多模态 AI 有兴趣，欢迎在评论区留言讨论，或者直接试着把上面的思路跑一遍，看看能不能在你自己的项目里也搞出一个虚拟家具 staging 的小功能。我们下期再见！

【描述】
最近看到很多开发者在尝试用多模态大模型做虚拟家具 staging，觉得一次调用就能拿到图片、文案和语音。其实这种“一键搞定”的想法在 Gemini 3.1 Flash 上会碰到模型能力边界，导致超时和失败。今天的文章拆解了作者在 Gemini Live Agent Challenge 中的实战过程，展示了如何用合适的模型分步完成任务，避免盲目追求一次性输出。

核心思路是先用 sharp 把上传的房间照片压缩到合适大小，再送给 gemini-3.1-flash-image-preview，只让它负责在房间里添加家具、调整光影和材质；随后把生成的 staged 图片的 base64 再交给 gemini-2.5-flash，产生房产描述和风水建议；最后调用 Google Cloud Text-to-Speech 生成 MP3 音频。整个流程在 Cloud Run 上通过六十秒超时和重试机制包装，确保偶发网络抖动不会导致实例频繁重启。

这样拆解带来的直接好处是：图像模型专注像素生成，文本模型擅长语言理解，避免了等待不存在的文本响应而占用连接。开发者可以把这三步封装为独立的 Cloud Run 服务或无状态函数，按需伸缩；中间结果（压缩图片、生成文本）可暂存到内存或 Supabase，便于出错时快速回滚，节省算力和时间。适用于房产中介快速出效果图、短视频配音以及线上看房的交互场景。

亮点在于作者把“一次请求搞定”的幻想拆解成了明确的模型职责分工，并给出了具体的重试、超时和日志策略；局限在于目前仍依赖于 Gemini 的 preview 版本和特定提示词，生成家具的尺寸和风格还有提升空间。从趋势看，多模态应用正朝着“专模型+编排”方向发展，单一巨模型虽然强但往往在特定任务上不如专门模型高效。

如果你也想试虚拟 staging，可以直接克隆作者的思路：先压缩图片，再分别调用 Gemini 图像、文本和 TTS 接口，记得加上超时和重试。欢迎在评论区分享你在多模型编排中遇到的坑，或者告诉我你想把哪些其他模型串起来做什么有趣的应用！
