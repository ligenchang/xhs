【标题】
Harness 是否真是 AI 工程的核心

======================================================================
【正文】
最近在看一些关于 Agent Engineering 的讨论，一直被一个词反复提到——Harness。其实以前在做金融的时候也经常琢磨同样的问题，就是当一个人能赚到三百万的时候，这三百万到底有多少是他个人的能力，有多少是他所在的岗位、平台或者品牌带来的。现在在 AI 领域，同样的争议出现在 Harness Engineering 上，大家在争论到底是大模型本身的能力更重要，还是围绕模型构建的那套系统、工具链更关键。

先说说什么是 Harness。按照我在和一些工程师聊天时得到的共识，Harness 在任何工程学科里都指的是那层负责连接、保护和编排各个组件的东西，它自己不做实际的计算工作，但却是让整个系统能够跑起来的基础。比如在传统软件里，框架、中间件、甚至就是一个简单的 main 函数都可以算是一种 Harness。在 Agent 领域，Harness 就是那个把模型调用、工具执行、上下文管理串起来的循环。

最近看到几段播客讨论，Boris Cherny 和 Cat Wu 在讲 Claude Code 的时候一直强调它的 Harness 有多么简单。Boris 说基本上没有什么秘密的源码，全是 JavaScript，反编译一下就能看到，他们一直在做的就是把模型包装得尽可能薄，几乎可以说是模型的最外层包装。Cat 也说他们故意保持最简单，甚至每隔几周就会重写一次，就像那艘这些斯船，零件不断被换掉但船还是那艘船。这种说法让人觉得，也许他们觉得自己做的东西根本不算什么技术壁垒，甚至有点担心自己是不是被模型方给边缘化了。

反过来看 OpenAI 那边的表态，Noam Brown 在一次分享里提到，在推理模型出现之前，大家为了让模型表现出类似推理的行为，其实做了很多工程工作——比如让 GPT-4o 多次调用、加入各种复杂的调度逻辑。但现在有了真正的推理模型以后，这些复杂的包装似乎反而变得没那么必要了。他甚至说，未来可能只需要一个统一的大模型，根本不需要上面再额外加一个 Router 或者其他的调度层。这种观点让人觉得，也许 Harness 只是一个过渡阶段的产物，等模型能力足够强的时候，这些中间层会自然被吸收进去。

不过也有完全不同的声音。有些工程师坚持认为 Harness 才是产品的核心。他们指出，不管是 Claude Code、Cursor 的 agent 还是 Manus，其实它们的核心循环就只有这么几步：模型返回工具调用 → 执行工具 → 把结果塞进上下文 → 再次调用模型。这个 while 循环看起来简单，但真正落地的时候，涉及到工具的安全隔离、结果的缓存、上下文长度的控制、失败时的重试策略等等，这些细节决定了整个系统能不能在生产环境里稳定运行。Jerry Liu 曾经说过，从模型中获得价值的最大障碍其实是你自己对上下文和工作流的编排能力，换句话说，如果不会玩 Harness，即便模型再强也发挥不出作用。

我在实际做一些小实验的时候也有类似的感受。比如用一个通用的 SWE-Agent 框架跑一下 Opus 4.6 和 GPT-5.2 的表现，发现有时候同样的模型在不同的 Harness 下会有几个百分点的差距，而这个差距在误差范围内来回摇摆。这其实说明，有时候 Harness 的选择并不是决定性因素，但也不能说完全没关系。当你把模型换成更强的版本的时候，原来那套看似臃肿的 Harness 可能会变得多余；而当模型还不够成熟的时候，一个好的 Harness 反而能让你少走很多弯路。

说到底，这就像是当年讨论“人 versus 岗位”时的结论——两者都重要，只是在不同的阶段权重会变化。当模型还在快速迭代、能力还有明显短板的时候，投资在一个稳固、易于扩展的 Harness 上往往能让团队更快看到产出；当模型达到某个临界点，能够自处理更复杂的任务时，过度的包装反而会成为负担。于是我们看到现在的研究方向其实也在往两边走：一边是尽量让 Harness 越来越薄，甚至想把它完全内嵌到模型的推理过程中；另一边则是继续打磨 Harness 的可靠性、可观测性和可定制性，让它在大模型还不能完全胜任所有场景的时候，能够提供足够的保障。

写到这里我也想到自己的一些小困惑。比如有时候我看到一些开源项目声称自己的 Harness 只有几十行代码，实际上在使用的时候还得额外引入一堆依赖、配置文件，甚至还要写一些胶水代码来适配不同的工具链。这时候到底算不算“最薄”的 Harness 呢？还有就是当我们谈论“单一统一模型”时，是不是真的意味着未来不需要任何外部调度？还是只是说我们希望模型自己能够学会调用工具、管理上下文，从而把这些职责内化进去？

这些问题其实没有标准答案，就像当初那个关于交易员的争论一样，最终还是要看具体的情景和目标。如果你是在做一个需要快速验证想法的原型，也许一个极简的 Harness 足够；如果你是在构建一个要面对真实用户、需要长期稳定运行的产品，那么投资在一个经过思考的 Harness 上可能会更安心。不管怎样，保持对两边都有了解，才能在技术演进的路上不至于被一方的观点牵着走。

希望这番唠叨能给正在纠结 Harness 到底是不是真实存在的同仁一些参考。如果你也有自己的实践经验或者不同的看法，欢迎在评论区聊聊，毕竟技术的进步本来就是大家一起碰撞出来的。祝大家 coding 开心，别让 Harness 或者 Model 给绊住了脚。

【描述】
最近在看 Agent Engineering 的讨论，Harness 一词被反复提到。它看似只是把模型、工具、上下文串起来的胶水层，却直接决定了 AI 应用能否在生产环境里跑通。对于正在尝试用大模型构建产品的开发者来说，现在正是评估这一层到底值不值得投入的时候。

文章指出，Harness 在任何工程学科里都是连接、保护和编排组件的基础层，它本身不做计算但让系统能够运行。在 Agent 领域，它负责模型调用、工具执行和上下文管理的循环。争议的焦点在于：当模型能力还不够强时，一个稳固、易扩展的 Harness 能大幅提升产出；而当模型达到临界点后，过度的包装反而会成为负担。

从实用角度看，一个好的 Harness 能帮你快速验证想法：只需几行代码把模型包装起来，就能跑通工具调用和上下文传递。当需要面对真实用户时，它还提供安全隔离、结果缓存、重试策略等细节保障，让系统在高并发或异常情况下保持稳定。适用于原型探索、内部工具以及需要长期运营的 AI 产品。

不过，Harness 也有局限：过度依赖它可能掩盖模型自身的不足，导致团队在模型能力提升时迟迟不愿升级。此外，有些声称只有几十行的开源 Harness 实际上还需要大量依赖和胶水代码，真正的“最薄”难以实现。未来趋势是两条路径：一是把 Harness 内嵌到模型推理过程，二是继续打磨其可观测性和可定制性，以在模型尚未全能时提供保障。

如果你正在构建基于大模型的 Agent，不妨先用一个极简的 Harness 跑通基本流程，再根据实际表现逐步加入安全、缓存和监控功能。欢迎在评论区分享你的实践经验或疑问：你认为在什么情况下 Harness 才是真正的瓶颈？现在就来聊聊吧。
