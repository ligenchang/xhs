从零实现KVCache提升LLM推理效率

======================================================================
【正文】
最近在写点儿 LLM 推理的小玩意儿，顺手聊聊 KV cache。这玩意儿在生产环境里是提升速度的关键，下面把概念和点儿代码写出来，算是给自己做个笔记。

先回顾一下生成过程。假设模型已经看到prompt「Time」，它会一步步产生下一个 token，比如「flies」，随后又基于「Time flies」继续推理。每一次都要把之前的所有 token 重新跑一遍注意力，这会导致大量重复计算。

注意力机制本质上是把所有 token 的 embedding 投射成 key、value、query 三个向量。每个 token 对应的 key、value 会参与相似度计算，得到注意力权重。如果不缓存，每生成一个新 token，模型就必须把整个序列重新编码，导致计算量呈线性增长。

图示的话，想象一下「Time」和「flies」已经生成，接下来要生成「fast」。如果不使用缓存，模型要重新处理「Time」和「flies」两个 token，才能得到对应的注意力输出。这种冗余在序列变长时会越来越明显。

KV cache 的核心思想就是在推理阶段把这些 key、value 保存下来。当模型已经算好某个位置的 key、value，后面的解码步骤只需要查询一次，而不是每次都重新计算。这样就能把注意力的复杂度从 O(n²) 降到 O(n)，显著加快生成速度。

具体来说，解码第 t 步时，模型只需要取出已经缓存的前 t‑1 个 key、value，和当前的 query 进行注意力运算。这样既省掉了重复的线性映射，又保持了上下文信息的完整性。

下面给出一个最简版的实现思路，用 Python 伪代码演示。假设已经得到 token_embeddings，然后通过线性层得到 q、k、v。这里我们只缓存 k 和 v，存到列表里

初始化缓存
kv_cache = []
for step in range(max_len):
    x = token_embeddings[step]
    q = linear_q(x)
    scores = q @ k.T / sqrt(d_k)
    这里省略 softmax 等细节
    kv_cache.append((k, v))

生成时只需要把 kv_cache 中的 k、v 拼起来参与矩阵乘法即可。

需要注意的是，缓存的 key、value 维度随层数和头数而增长，实际项目里往往把它们拼成一个大的张量。内存占用会随序列长度线性增长，所以在超长上下文场景下要注意控制上下文窗口，或者采用分段切分的方式。

从实用角度看，KV cache 只能在推理阶段使用，训练阶段依然需要完整的序列计算。另外，因为缓存结构增加了代码复杂度，新手在实现时容易漏掉一些边界条件，比如长度截断、mask 处理等。

不过在实际部署中，很多框架已经把 KV cache 抽象成统一的接口，开发者只需要关注输入的 query、key、value 三个张量的形状即可。配合流式推理引擎，能够在毫秒级别完成几千 token 的生成。

总的来说，KV cache 之所以重要，是因为它把注意力的重复计算降到最低，同时只带来线性内存开销。只要合理控制上下文长度，配合高效的存储结构，就能在实际业务里获得数倍的吞吐提升。

希望这篇随手记能帮到正在研究 LLM 推理的同学，后续如果有机会再深入聊聊多头注意力的实现细节，就不一味儿说了。

在实际部署时，KV cache 的大小往往占据显存的很大一部分。比如一个 12 层、8 头的模型，每生成 1000 个 token，缓存的 key、value 大约需要几百兆字节的显存。为了避免显存爆炸，很多工程会采用分页或滑动窗口的方式，只保留最近 N 个 token 的缓存。这样既能保持上下文，又能控制显存峰值。
