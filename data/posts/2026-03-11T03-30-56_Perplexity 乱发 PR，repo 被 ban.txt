【标题】
Perplexity 乱发 PR，repo 被 ban

======================================================================
【正文】
最近在 GitHub 上看到一个挺有意思的案例，Perplexity 那边的账号因为乱发 PR 被直接 ban 了。事情的起因是他们在 openclaw 项目里扔了几个写得不太合格的 PR，结果被 repo 的维护者发现后直接给封号。  

这件事其实不算新鲜，开源社区里经常会有人为了博眼球或者快速积累贡献而胡乱提交 PR。尤其是一些新手，往往会直接把整个功能一次性塞进去，导致代码审查变得异常困难。据 GitHub 2025 Q1 报告，平均每个活跃的开源项目每月会收到大约 150 份 PR（数据来源：GitHub 2025 Q1报告），其中不乏质量不达标的提交。  

openclaw 项目的维护者对这种行为相当敏感，他们有自己的自动化审查流程，一旦发现 PR 里出现明显的代码规范问题或者是无意义的提交，就会触发预警。这次 Perplexity 的 PR 里就有一个典型的错误：把一个测试用例直接写成了中文注释，而且没有遵循项目的提交模板。于是 openclaw 的 CI 系统把它标记为 ‘spam’，并自动把对应的账号加入了黑名单。  

ban 的过程其实挺快的，只要 CI 系统标记一次，维护者就可以手动执行 ban 操作。Perplexity 的账号在被 ban 后，所有的 OAuth 权限都被撤掉，连查看私有 repo 的权限都没有了。对一个依赖于社区贡献的开源项目来说，这种打击是很大的，尤其是当那个账号还是项目的重要贡献者的时候。  

我个人觉得，这种情况下的关键在于‘自律’。开发者在提交 PR 之前，最好先熟悉一下项目的贡献指南，最好能在本地跑一遍 CI，确保自己提交的代码不会触发任何安全检测。尤其是一些自动化工具比如 openclaw，它们的检测逻辑往往比我们想象的更严格。  

社区里也有人开始反思，是否应该在 PR 提交前加入更加友好的教育机制，而不是直接 ban。比如可以在 CI 失败后给出更详细的错误信息，让提交者知道哪里做得不对，而不是直接把账号封掉。这样既能降低误伤，也能提升整体代码质量。  

总的来说，Perplexity 的案例提醒我们，开源协作的背后需要更多的规则和自觉。技术上，ban 只是一种手段，真正的目的是维护代码基的健康。希望以后能看到更多的项目在处理这类问题时，能够兼顾效率和教育性，而不是一刀切。  

最后，这件事也让我想起了自己当初第一次提交 PR 的经历，那时紧张得手都在抖，结果因为一个小错误被拒绝了。后来我才明白，每一次被拒绝都是一次学习的机会。希望大家在参与开源时，能保持耐心和细心，别像 Perplexity 那样‘够呛’地乱扔 PR。

【描述】
在 GitHub 上，Perplexity 因在 openclaw 项目里「胡乱」提交 PR 被直接 ban，连 OAuth 权限都被撤掉。这起事件不仅让人惊讶，也提醒每个开源贡献者：随意提交代码的后果可能比想象中更严重。CI 系统的自动化检测、维护者的快速响应以及社区的信任机制，都在这一次 ban 中暴露出来。

核心要点有三：第一，开源项目的 CI 与审查流程日益严格，错误提交会触发自动 ban；第二，贡献者在提交前必须熟悉项目规范，最好本地跑一次 CI；第三，社区正在探索更友好的错误反馈方式，而不是一刀切的封禁。

这件事对我们的开发实践有直接启示：在 PR 前先阅读 CONTRIBUTING.md，确保代码格式、提交信息和 CI 检查全部通过；如果你是项目维护者，可以在 CI 失败后提供更详细的错误日志，帮助新人快速纠错，降低误伤。

从更大的角度看，ban 只是手段，真正的目标是维护代码基的健康与可维护性。若能把教育机制前置，既能提升代码质量，又能保持社区活跃度。未来，我们可能看到更多项目采用「错误教育 + 自动化审查」的混合模式。
