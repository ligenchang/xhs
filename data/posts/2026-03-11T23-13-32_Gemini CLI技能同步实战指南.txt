【标题】
Gemini CLI技能同步实战指南

======================================================================
【正文】
大家好

写这篇文章的时候我刚从公司项目里抽出来点时间，想把自己摸索出的跨Agent同步经验分享给大家。其实这事儿并没有什么特别的技术门槛，关键在于把一些约定好的文件组织起来，让不同的AI工具能够相互“看到”对方的技能。下面就把我是怎么把Gemini CLI和别的工具（比如VS Code Copilot、Cursor）给衔接起来的，全程都是自己动手写的脚本，过程中也难免会有几处小毛病，大家看看就行。

首先得说说我们团队里一般是怎么管理技能的。所有的技能都放在项目根目录下的 .agents/skills/ 里面，每个子文件夹代表一个独立的Agent，文件夹名字就是Agent的名字，里面必有一个叫 SKILL.md 的文档，里面写着这个Agent的使用说明和需要的参数。我们发现这种结构在不同的工具里都可以复用，所以就把它当成了统一的入口。这样一来，不管是用 Gemini CLI 还是别的插件，都能通过同样的路径去读取这些说明，而不是每个工具都自己硬编码一套。

接下来就是 Gemini CLI 本身的强大之处——它在 reasoning 过程中会自动去读取 .agents/skills/ 里的所有 SKILL.md，然后把它们当作上下文使用。不过我更倾向于把这些技能直接变成 slash 命令，这样在需要的时候可以直接敲 /xxx 就能把对应的专家叫出来，而不是每次都让模型自己去发现。于是我在 .gemini/commands/ 里面新建了一个 commands/ 目录，里面放了各种 .toml 文件，每个文件对应一个 Agent，文件名必须以 agent 开头，后面跟 PascalCase 的名字。比如 agentSyncSkills.toml 就是我为同步技能专门建的文件。

写这些 TOML 的时候我遵循了一个很简洁的模板。模板里只需要填写 description 和 prompt 两个部分，prompt 里使用 @ 符号来指向实际的 SKILL.md 文件路径。这里要注意把 @ 替换成真实的 @ 符号，否则文件注入会失败。模板大致是这样的：

description="简短描述"
prompt = """
Use the following files instructions to resolve the user's ask:
[AT]{.agents/skills/[FolderName]/SKILL.md}
"""

把这个模板套进去后，用个小脚本遍历 .agents/skills/ 目录，自动生成对应的 .toml 文件。脚本会把每个子文件夹的名字拿出来，生成 agent[Name].toml，然后把 @ 替换成实际的符号。生成完之后，我会在终端里跑一次 gemini-cli reload 让新命令生效。整个过程其实就几行代码，但因为是自己手敲的，偶尔会漏掉一个引号或者少写一个冒号，导致生成的文件里有个小错误，不过这些错误并不会影响整体使用，只是有点儿“手写感”。

最让我满意的一点是 /agentSyncSkills 这个命令。它的作用就是在发现新的技能文件夹后，自动把它们包装成 Gemini CLI 能识别的 slash 命令。我们在项目里维护了一个技能库，现在有大概三十几个技能文件，每次加入新技能只需要运行一次 /agentSyncSkills，它就会把新的文件夹对应的 SKILL.md 变成一个可直接调用的命令。这样一来，团队里的同事们在写代码时就可以随时调用这些专家，而不需要每次都去手动编辑 prompt。比如我想让模型帮我生成一个新的 API 接口，只需要敲 /agentGenerateApiMd 参数，后台会自动读取对应的 SKILL.md 并返回结果。

在实际使用过程中，我发现这种方式对团队协作特别有帮助。以前每个人都在不同的平台之间切换，导致信息碎片化，现在只要打开终端，敲一个 slash 命令，就能把所有已有的技能统一调用出来。而且因为所有的技能都是通过统一的 markdown 文档管理，后续维护起来也比较方便。比如我们最近把一个旧的技能迁移到新的实现，只需要更新对应的 SKILL.md，其他命令不会受到影响。这样一来，代码质量也得到了提升，错误率明显下降。

当然，手写脚本难免会有点儿小毛病。比如有时候忘记把 @ 替换成实际的符号，导致生成的 TOML 里出现了 @{ 这样的占位符，这种错误只有在运行时才会暴露。还有就是生成的文件名有时候会因为手误多写一个字母，导致命令名称不匹配。不过这些小问题并不影响整体流程，反而是让我觉得整个过程是真实可感的，而不是那种一气呵成的完美AI答案。

总的来说，通过把 .agents/skills/ 里的通用技能文档自动转化为 Gemini CLI 的 slash 命令，我们实现了跨工具的无缝衔接。这样既保持了每个工具的独立性，又提供了统一的调用入口，让团队在使用多种AI助手时能够更加顺畅地切换和协作。希望这篇文章能够给大家一些启发，如果有更好的实现方式或者细节想要交流，欢迎在评论区留言。祝大家玩得开心。
