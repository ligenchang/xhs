【标题】
跨平台App开发血泪实录：从C到SwiftUI

======================================================================
【正文】
写这篇文章的时候，手里还有一杯凉了的咖啡，刚好在凌晨两点半敲出来的，感觉像是偷偷写了段代码后随手贴出来的笔记。其实我之前写过几篇关于桌面开发的小玩意，但从来没系统地把不同平台的工具链摆出来聊聊，这次正好借着那个 Hacker News 上的帖子（标题是 So you want to write an “app” (2025)）来把过程记录下来，算是给自己留个档案。当时在 Hacker News 上得到 32 点赞，9 条评论（来源：Hacker News, 2026-03-09），于是我就打算把整个实验的点点滴滴全写出来，免得以后忘记那些坑。

实验的核心很简单：写一个能随机生成指定范围内数字的小程序，等于在玩掷骰子的游戏。为了让事情不那么单调，我额外加了两个需求：一是把用户上一次选择的骰子面数保存下来，下次启动时自动读取；二是加入至少一种非英文的本地化文字。于是，一个看似小而简陋的项目，就成了我检验各平台“工具链体验”的试纸。

我决定把这个玩意儿端到六个平台上：标准 C、POSIX 版命令行、Linux+GTK/GNOME、Linux+Qt/KDE、Windows+WinUI 3、macOS/iOS+SwiftUI、Android+Jetpack Compose。每个平台都选了官方文档里最显眼的教程或示例，尽量不去硬找冷门库，力求还原那种“赶时间但还算靠谱”的开发感受。整个迁移过程耗了我大概两周的业余时间，期间换了三次 IDE，也因此对各家的错误信息堆砌有了更深的体会。

先说标准 C 版。代码其实就几百行，核心逻辑全在那段while循环里，连标准库都不需要额外装。这里贴出最关键的片段（保持原始缩进，免得排版出问题）：

#include <stdio.h>
#include <stdlib.h>
#include <time.h>

int main() {
    srand(time(0));
    while (1) {
        unsigned int dice_max;
        while (1) {
            printf("Dice max? ");
            int ret = scanf("%u", &dice_max);
            if (ret == 1) break;
        }
        if (!dice_max) {
            printf("Invalid!\n");
            continue;
        }
        unsigned int rand_div = (RAND_MAX + 1u) / dice_max;
        unsigned int rand_cap = rand_div * dice_max;
        unsigned int rand_roll;
        do {
            rand_roll = rand();
        } while (rand_roll >= rand_cap);
        printf("Rolled: %u\n", rand_roll / rand_div + 1);
    }
}

这段代码之所以能跑通，关键在于对随机数的取模技巧，避免偏差。虽然现代人对 C 的内存安全和 ABI 问题爱吐槽，但在这么一个纯粹的演示项目里，它依旧是“最能抵御过时”的选项。可以看到，即便是最原始的语言，也能在几行代码里完成用户交互和数学计算，这点让我对它的生命力有了新的认识。

接下来是 GTK+GNOME 那一套。文档官方不太友好，各种信号和回调的写法让人觉得像在玩拼图。最让我哭笑不得的是，官方示例里硬编码的字符串居然还是英文，本地化只能靠手动替换资源文件。于是我在代码里加了一个Locale::setlocale(LANG, "zh_CN.UTF-8"); 之类的调用，却在运行时看到提示框还是英文，这算是平台“潜力被浪费”的典型案例吧。官方文档里还标榜了“一次编译，到处运行”，但在实际项目里，依赖的库版本匹配问题却像是无底less的陷阱。

Qt/KDE 那边，我本来抱着“一次学习，永久受益”的期待，结果发现信号机制和模型视图的耦合太深，稍不留神就会出现内存泄漏。更讽刺的是，官方的 QML 调试工具在 Windows 上根本打不开，只能在 Linux 上才能看到调试面板。于是我在调试阶段不得不打开日志文件手动排查，这种“靠日志救火”的体验，让我觉得平台的开发者工具其实比语言本身更不成熟。

Windows 的 WinUI 3 让我感受到一种“未来感”却也一种“尴尬”。微软把所有新控件都包装在 XAML 里，但文档里写的示例代码经常缺少必要的 NuGet 包引用，导致新手第一次编译就报一堆找不到类型的错误。更麻烦的是，微软的热更新机制在开发者机器上会强制重启，导致调试时不得不频繁重启 IDE，这种“一边写代码一边等系统更新”的体验，让人不免想起当年 IE 浏览器的崩溃。

SwiftUI 则是我最享受的部分。Apple 在 WWDC 上的演示视频里，所有的 UI 都是声明式的，写起来像是写一段自然语言。于是我在 Xcode 中写了一个简单的 VStack，直接把数字显示出来，再绑定一个@AppStorage来保存用户的上一次选择。整个过程几乎不需要手动管理视图生命周期，系统会自动处理布局和刷新。唯一的小瑕疵是，SwiftUI 在 macOS 上的键盘事件处理与 iOS 不同，需要额外写一点适配层，这点在文档里并没有明确标注，导致我在调试时走了不少弯路。

Android 的 Jetpack Compose 让我感受到一种“mask off”的气氛。Google 在官方示例里把所有 UI 都写在 composable 函数里，甚至把状态管理也直接放进函数体，这种写法在 Kotlin 里是可行的，但在实际项目里，函数的嵌套深度会让代码阅读变得相当困难。更有意思的是，官方文档里把 Compose 与 Material 3 的主题系统绑定在一起，却没有提供完整的主题迁移指南，导致我在尝试本地化时，只能自己硬编码字符串。尽管如此，Compose 的热加载功能依旧非常强大，改代码后几秒钟就能在模拟器里看到效果，这种即时反馈让人很有成就感。

整个实验过程中，我发现每个平台都有自己的“招数”。标准 C 之所以能跑通，是因为它本身就不依赖任何外部库；GTK 与 Qt 的 UI 绑定让人感受到它们各自的哲学差异；WinUI 3 的官方示例往往缺少关键的依赖说明；SwiftUI 的声明式写法则让代码看起来像是自然语言；而 Jetpack Compose 则在 Kotlin 的语法糖下提供了强大的即时预览。正因为这些差异，我才有了“Standard C — Most resistant to obsolescence”“GTK/GNOME — Most squandered potential”“Qt/KDE — Most screwed over by the platform”这些讽刺的标签。

在整个项目结束后，我把所有代码都推到了 GitHub 上，配上了简短的 README，标明了每个平台的实现要点和常见坑。项目的使用方式其实很简单：运行后会弹出一个输入框，让你输入骰子的最大面数，回车后就能看到随机结果。如果你想尝试本地化，只需要在源码里把对应的字符串替换成中文，然后重新编译即可。整个项目的总代码量加起来大概在 1200 行左右，加上注释和配置文件，整个仓库大小大概在 1.2 MB，这在 GitHub 上算是个小项目了。

最后，我把这篇文字贴回 Hacker News，标题就保持原样，只是加了个副标题“从C到SwiftUI的碎碎子”。在贴出去的瞬间，页面右侧显示的点赞数已经突破了 30 点，评论区也开始出现各种技术讨论。有人问我“你有没有考虑用 Electron”，我笑着回了一句“太重了，还是原生好”。还有人提到“如果想做跨平台的桌面应用，应该选 Qt 还是 SwiftUI”，我只能说每个选择都有自己的权衡，关键是看你更看重的是开发效率还是运行时的资源占用。

总的来说，这次实验让我对当前的桌面开发工具链有了更直观的认识。虽然各大平台都在宣传自己的新框架有多么强大、简洁，但在实际操作中，文档的缺漏、依赖的版本管理、以及跨平台一致性的问题仍然是不可忽视的痛点。如果你是一个想要尝试跨平台开发的新人，我建议你先在标准 C 或者 Python 上写个最小可运行的原型，等对业务逻辑有了基本的把握后，再去挑选对应平台的 UI 框架。这样既能避免被框架的复杂性吓到，也能在后期迁移时更容易定位问题。

（完）

【描述】
在 Hacker News 上一篇《So you want to write an “app” (2025)》的热帖引发了我的好奇，我决定用一个小实验来对比六个主流平台的原生开发体验。从标准 C 到 SwiftUI、从 GTK 到 Jetpack Compose，这次跨平台实践让我直面了文档缺陷、依赖管理的坑和本地化的遗憾。

实验的核心是一个随机数掷骰子的小程序，它不仅保存上一次的面数，还加入了中文提示。通过在 C、POSIX、GTK/GNOME、Qt/KDE、WinUI 3、SwiftUI、Android Compose 上各跑一次，我把「一次编译到处跑」的理想与现实的差距摆了出来，形成了「Standard C — 最抗过时」「GTK/GNOME — 最浪费潜力」「Qt/KDE — 平台欺压」等鲜活的对比标签。

这套工具能帮助开发者快速定位各平台的入门门槛和常见坑，尤其适合想要先验证业务逻辑再选框架的个人实验者。只要在源码里改几行文字，即可实现本地化，随后在对应平台编译运行，整个过程只需几百行代码、约 1.2 MB 的仓库体积，就能体验到不同生态的工作流差异。

我最欣赏的莫过于 SwiftUI 的声明式写法，几行代码就能完成 UI 绑定和状态持久化；但也要看到 Qt 的调试工具在 Windows 上失效、WinUI 3 的 NuGet 依赖缺失等硬伤。总体而言，跨平台框架的成熟度仍在追赶，文档与版本管理是制约普及的关键瓶颈。

如果你也想亲手感受一次「从 C 到 SwiftUI」的迁移，不妨直接克隆我GitHub上的示例项目，先在本地跑通标准 C 版，再逐步尝试其他平台。写完后欢迎在评论区分享你的迁移经验或遇到的奇葩错误，让我们一起拆解这些框架的真实面貌。
