【标题】
容器AI助手安全指南：nanoclaw

======================================================================
【正文】
最近在 GitHub 上看到一个挺有意思的项目，叫 qwibitai/nanoclaw，号称是 OpenClaw 的轻量替代，运行在容器里，安全性更高。星星数已经突破 21625 星，最近更新是 2026-03-11。项目的核心卖点就是把 AI 助手放进自己的容器里，既能保证隔离，又能直接对接 WhatsApp、Telegram、Slack、Discord、Gmail 这些常用聊天工具，还支持内存、调度任务，直接基于 Anthropic 的 Agents SDK 搭建。嗯，感觉挺实用的。

和 OpenClaw 不同，后者代码基线差不多有半十万行，配置文件多达五三个，依赖也有七十多个，安全只做了应用层的白名单和配对码之类的检查，而不是真正的操作系统级别隔离。整个系统全跑在一个 Node 进程里，共享内存，稍有不慎就可能把整个环境搞崩。而 nanoclaw 把这一切压缩到只有一个进程、几个源码文件，连依赖都尽量控制在最小范围。更重要的是，它把每个 Agent 放进独立的 Linux 容器里，文件系统只挂载需要的目录， Bash 访问也只能在容器内部执行，host 系统不受影响。这样一来，安全性提升了不少，尤其是在处理敏感数据时，容器的隔离能够防止意外泄漏。

我个人最喜欢的点是它的哲学：小到能完整看懂。整个代码库只有一个进程，几个源码文件，没有微服务拆分，想彻底弄清楚它的实现只要跑一句 Claude Code 让它走一遍代码路线就行。安全上，它不靠权限检查，而是靠容器隔离，每个 Agent 只能看到自己挂载的文件系间，连 Bash 都只能在容器里敲命令，host 完全不暴露。其实有时候写代码的时候会忘记加注释，但因为代码本身不长，阅读起来还是挺顺手的。

另一个亮点是它支持 Agent Swarm，也就是可以在聊天里一次性 spawn 多个专门的 Agent，让它们协作完成复杂任务。这在个人 AI 助手领域里还是首次，我之前只在实验室里看到过类似的概念，现在居然能直接在自己的机器上玩起来。嗯，感觉挺新奇的。

快速上手的步骤也挺直白的：首先用 GitHub CLI fork qwibitai/nanoclaw --clone，或者直接在 GitHub 页面点 Fork 按钮，然后 git clone 你的 fork 地址，进入目录，运行 claude 命令。如果没有 GitHub CLI，就手动在 GitHub 上 Fork，然后 git clone https://github.com/<your-username>/nanoclaw.git，cd 进去，再敲 claude。随后在 claude 提示符下输入 /setup，Claude Code 会自动处理依赖、认证、容器搭建和服务配置。这里要注意，所有以 / 开头的指令（比如 /setup、/add-whatsapp）都是 Claude Code 的技能，必须在 claude CLI 里输入，而不是在普通终端里执行。如果还没装 Claude Code，可以去 claude.com/product/claude-code 下载。其实第一次运行时会有点小卡顿，但很快就好了。

整个流程其实挺顺手的，尤其是 Claude Code 能自动搞定容器的初始化、密钥配置、权限挂载，省去很多手动折腾。而且整个过程没有图形化的安装向导，没有监控面板，你只能通过聊天的方式问它‘现在在干啥’，或者直接描述问题让它帮你调试。感觉挺有个性的，不过也挺符合开发者的习惯。

从设计理念来看，nanoclaw 更倾向于个人用户，而不是大企业级别的框架。它不追求功能堆砌，而是鼓励用户自己 fork 一份，让 Claude Code 去改代码，把功能调到最符合自己需求的状态。这样一来，代码库保持精简，改动也安全可控。想要不同的行为，就直接改源码，而不是去配置一堆参数。这种 ‘code is config’ 的思路在 AI 时代显得特别直观，因为大模型本身就擅长生成代码，所以你可以让它帮你写新的技能，比如 /add-telegram、/add-discord 之类的插件，最终得到的代码库还是那几百行，但功能却能无限扩展。嗯，挺有意思的。

运行在 Claude Agent SDK 上也是一个关键点，意味着你使用的其实是 Claude Code 本身，它在编码和问题解决能力上相当强，能够自动修改、扩展 nanoclaw，甚至针对每个用户的需求定制化。这样一来，你得到的不是一个通用的框架，而是一个专属的 AI 助手。项目里提到的多채道消息支持，让你可以同时在 WhatsApp、Telegram、Discord、Slack、Gmail 里和助手对话，每个渠道都可以通过对应的技能加入，比如 /add-whatsapp。每个群组都有独立的 CLAUDE.md 内存、独立的文件系统、独立的容器沙箱，只有挂载的目录才能被看到。还有主通道，也就是自我聊天，用来做管理员控制，所有群组互相隔离，互不干扰。这样管理起来更清晰。

调度任务也很实用，可以设定周期性的任务让 Agent 自动执行，比如每天凌晨给你发一条天气预报，或者每周总结一次项目进度。还有网络访问能力，能够在运行时去搜索互联网获取最新信息，这在需要实时数据的场景下非常有用。每个 Agent 都有自己的记忆文件，叫 CLAUDE.md，里面可以写下你想让它记住的上下文，这样在后续对话中它会引用这些信息，形成连贯的对话体验。调度任务可以写成类似 /add-cron '0 9 * * * 发送天气' 的形式，Claude Code 会把它变成实际的 cron 任务，定时触发，甚至可以让 Agent 把结果发回聊天，让你在闲时看到。嗯，挺方便的。

通过内置的 web fetch 能力，Agent 能在运行时打开网页、抓取数据，甚至解析 JSON，把外部信息直接喂给模型，实现更智能的决策。虽然项目本身不提供监控面板，但社区里已经出现了不少分享的技能，大家在 GitHub 上提交自己的 /add-xxx，互相学习。因为代码本身很小，新人也能快速看懂并贡献自己的技能，形成良性循环。不过容器在 macOS 上的 Apple Container 可能不支持所有 Linux 功能，某些特定的系统调用可能需要额外配置。还有，如果你想在生产环境里长期运行，需要自行处理容器的持久化和日志收集，这部分还得靠自己写脚本。这些细节得自己摸索，但整体还是挺值得折腾的。

项目作者表示未来会继续完善 Agent Swarm 的协作机制，让多个 Agent 能更自然地分工合作，甚至可以在不同的容器之间传递消息。希望有一天能实现完全自治的个人 AI 团队，完成从计划到执行的全链路。总的来说，nanoclaw 给了开发者一种既轻量又安全的选择，让我们在不牺牲可控性的前提下，能够玩转大模型的能力。无论是想自己搭一个私人助手，还是想在聊天里组织一群专业的 Agent 协作，都能在这里找到合适的工具。嗯，感觉挺有前景的。

【描述】
最近在 GitHub 上看到一个叫 qwibitai/nanoclaw 的项目，它把 AI 助手完整封装进独立的 Linux 容器，实现安全隔离、资源占用极低，代码仅几百行，却能直接对接 WhatsApp、Telegram、Slack、Discord、Gmail 等常用平台。对喜欢轻量、可控的开发者，这无疑是一个值得深入探索的创新工具。

核心卖点在于：① 只跑一个进程、几百行代码，几乎不需要额外依赖；② 每个 Agent 独占容器，文件系统只挂载必要目录，连 Bash 都只能在容器内执行，host 完全不暴露；③ 通过 Claude Code 的 /setup 技能自动完成依赖、密钥和容器初始化，上手门槛极低。

这款工具能让个人用户在本地搭建专属的多渠道 AI 助手，支持定时任务、网络抓取和跨平台消息同步，尤其适合需要处理敏感数据、想自行掌控对话上下文的场景。开发者只需 fork 后运行 claude /setup，即可在几分钟内拥有一个可扩展、可定制的私有助手。

与传统的 OpenClaw 相比，nanoclaw 把安全从应用层白名单转向真正的容器隔离，代码可读性极高，改动只需直接编辑源码。它的哲学是“code is config”，让大模型本身成为配置工具，开启了 AI 代理自治的新可能，尽管在 macOS 容器兼容性上仍有限制，但整体思路已经领先。

如果你也想尝试把 AI 助手装进自己的容器，不妨先 fork 项目，跑一遍 claude /setup，体验一下多 Agent 协作的即时效果。在评论里告诉我你最想实现的自动化场景，或者遇到的任何配置难点，大家一起讨论、迭代。
